## Area: Security, IPC, build, tests, and ops

### Functional inventory

- **v1-archive (nanotars):** Security surface is concentrated in five host-side modules and one container-side module. (1) Mount allowlist at `src/mount-security.ts` reads `~/.config/nanoclaw/mount-allowlist.json` (path defined outside project root via `MOUNT_ALLOWLIST_PATH`) and validates each `additionalMounts` entry against allowed roots, blocked patterns, container-path traversal, and an `isMain`-aware RW gate (`mount-security.ts:232-343`); the `nonMainReadOnly` flag downgrades non-main RW requests to RO (`mount-security.ts:310-319`). (2) Outbound secret redaction at `src/secret-redact.ts` reads every value from `.env`, every `groups/*/.env`, and `~/.claude/.credentials.json`, subtracts a small `NON_SECRET_VARS` safe-list, escapes regex metacharacters, builds one composite regex, and exposes `redactSecrets()` (`secret-redact.ts:53-170`). It is wired through `routeOutbound()` (`router.ts:77`) and the container-runner's stdout/stderr log writes (`container-runner.ts:360-384`). (3) Auth-error detection at `router.ts:7-21` тАФ a seven-pattern `AUTH_ERROR_PATTERNS` array and an `isAuthError(text)` helper, consumed by the orchestrator (`orchestrator.ts:272-283`) and the scheduler (`task-scheduler.ts:216-226`) to emit a deterministic "[Auth Error]" message to the user instead of a silent retry loop. (4) IPC layer at `src/ipc/*` тАФ file-based per-group namespaces under `data/ipc/<groupFolder>/{messages,tasks,input}/*.json` polled by the host (`ipc/index.ts:21-124`). Identity is derived from the on-disk directory (`ipc/index.ts:48`), authorization is centralised in `ipc/auth.ts:6-35` (main-or-own-group), every read goes through `readIpcJsonFile()` with `O_NOFOLLOW`, lstat, and a 1 MiB cap (`ipc/file-io.ts:16-40`), and parse/auth failures are quarantined to `data/ipc/errors/` rather than deleted (`ipc/file-io.ts:58-62`). The discriminated union and type guards live in `ipc/types.ts:5-37`. Stdout protocol is a wrapped JSON in sentinel markers parsed by `output-parser.ts:34-89`, with per-run nonces (`makeMarkers`) to prevent agent-output injection. (5) Container hardening at `src/container-runtime.ts:155-177`: `--cap-drop=ALL`, `--cap-add=SYS_PTRACE`, `--security-opt=no-new-privileges`, custom `chromium-seccomp.json`, `--shm-size=2g`, `--init`, `--cpus=2`, `--memory=4g`, `--pids-limit=256`. (6) Bash security hooks at `container/agent-runner/src/security-hooks.ts:17-134` тАФ `createSanitizeBashHook()` denies `/proc/*/environ`, `/tmp/input.json`, `.credentials.json`, and read-tool combinations, then prepends `unset ANTHROPIC_API_KEY; unset CLAUDE_CODE_OAUTH_TOKEN` to the command; `createSecretPathBlockHook()` mirrors the same blocks on the Read tool. (7) Logging is `pino` + `pino-pretty` (`logger.ts:1-16`) with uncaught-exception/unhandledRejection trapping. (8) Tests are vitest only (`package.json:14-15`), 18 test files spanning `__tests__`, `ipc/__tests__`, `db/__tests__`, plus `container/agent-runner/src/security-hooks.test.ts`. (9) Build is npm + tsc, no supply-chain hold, no lockfile pin policy. (10) No GitHub Actions workflows present in `.github/workflows/` тАФ `ls /data/nanotars/.github/workflows` is empty.

- **upstream/main (nanoclaw v2):** Security surface is split across `src/modules/{mount-security,secret-redaction,group-env,approvals}/` plus the container-runner. (1) Mount allowlist at `src/modules/mount-security/index.ts` is a near-line-for-line port of v1's, with three deviations: `nonMainReadOnly` removed (`index.ts:23-30` schema doesn't carry it; `validateMount(mount)` has no `isMain` parameter), `matchesBlockedPattern` uses `part.includes(pattern)` instead of `part === pattern || part.startsWith(pattern + '.')` (`index.ts:152-170` тАФ wider matching, more false positives), and a new `:` rejection in `isValidContainerPath` to defend against Docker `-v` option injection like `repo:rw` (`index.ts:215-217`). The `nonMainReadOnly` semantics are gone because v2 doesn't have a "main group" concept. (2) Secret redaction at `src/modules/secret-redaction/index.ts` is also a clean port with two improvements: it sorts secret values by length descending before regex compilation so longer secrets always match before shorter prefixes (`index.ts:103`), and it adds `ONECLI_API_KEY` to `NEVER_EXEMPT` (`index.ts:32-38`). It also dedupes via a Set. There's a `secret-redaction.test.ts` covering 11 test cases. Wired through `delivery.ts:361` (`redactedContent = redactSecrets(msg.content)`); not used on stdout/stderr because the v2 container doesn't pipe stdout тАФ it writes to `outbound.db`. (3) Auth-error detection: **completely absent**. `grep -rn "isAuthError\|AUTH_ERROR\|oauth.*token.*expired\|invalid_api_key\|authentication_error" /data/nanoclaw-v2/src` returns zero hits. `delivery.ts` has no auth-aware error path; it just bumps `deliveryAttempts` and gives up after `MAX_DELIVERY_ATTEMPTS = 3` (`delivery.ts:33`). (4) IPC: replaced wholesale by per-session SQLite. `inbound.db` (host-write, container-read) and `outbound.db` (container-write, host-read) per session under `data/v2-sessions/<session_id>/`. The agent-runner polls `messages_in`, the host polls `messages_out` via `delivery.ts`. There is no file-based IPC, no JSON-files-on-disk, no `output-parser.ts` (deleted), no stdout markers, no per-group identity directory. The container does not write to its stdout for protocol тАФ it only writes app-level logs. (5) Container hardening: **all dropped**. `grep -n "cap-drop\|cap-add\|no-new-privileges\|--memory\|--cpus\|seccomp\|--shm-size\|--pids-limit"` in `src/container-runner.ts` and `src/container-runtime.ts` returns nothing. The only retained flags are `--rm`, `--user <hostUid>:<hostGid>` when host UID is unusual (`container-runner.ts:506-511`), volume mounts, and `--entrypoint bash` to source the env-dir before exec (`container-runner.ts:522-546`). `tini` is still in the Dockerfile (`container/Dockerfile:52`) as PID 1 for signal forwarding. (6) Bash security hooks at `container/agent-runner/src/security-hooks.ts:1-110` тАФ port of v1 with consolidated regex names (`PROC_ENVIRON_RE`, `INPUT_JSON_RE`, etc.), an expanded `READ_TOOLS_RE` that adds `more`, `od`, `hexdump`, `bun`, `awk`, `sed`, and `python3`, and `ONECLI_API_KEY` added to `SECRET_ENV_VARS` (`security-hooks.ts:23`). Tests are `bun:test` not vitest, with 10 cases (`security-hooks.test.ts:1-110`). (7) Per-group env passthrough at `src/modules/group-env/index.ts:1-138` is the explicit v2-side implementation: parses global `.env` and `groups/<folder>/.env`, applies an explicit per-group allowlist from `container.json:envAllowlist`, single-quote shell-escapes every value, writes the result to `data/env/<agentGroupId>/env`, and bind-mounts RO at `/workspace/env-dir` (`container-runner.ts:367-370`, `542-543`). The container then sources it via `set -a; . /workspace/env-dir/env; set +a`. (8) Credential model: OneCLI gateway at `src/modules/approvals/onecli-approvals.ts:1-269` and `container-runner.ts:486-500` тАФ every API call from the container goes via `HTTPS_PROXY` set by `onecli.applyContainerConfig(args, ...)` to the OneCLI vault, which injects credentials per request. The credential never sits in the container's env or filesystem. The vault is reached through a per-agent identifier (`ensureAgent({ name, identifier })`) keyed on the agent group id. Approval flow is a manual-approval handler that DMs an admin via `pickApprover` + `pickApprovalDelivery`. (9) Logging is hand-rolled `src/log.ts:1-65` тАФ no pino dep. ANSI-coloured, level threshold from `LOG_LEVEL`, separate stdout/stderr split at `warn`. Same uncaught/unhandled traps. (10) Tests: 23 vitest files in `src/`, 7 bun:test files in `container/agent-runner/src/`. The two trees can't share тАФ vitest can't load `bun:sqlite`; bun:test isn't pulled into vitest. **`src/modules/mount-security/` has no test file** (only `index.ts`), even though v1 ships `src/__tests__/mount-security.test.ts`. (11) Build is pnpm 10.33.0 + bun 1.3.12 split (host vs agent-runner), `pnpm-workspace.yaml:8` declares `minimumReleaseAge: 4320` (3-day supply-chain hold), `.npmrc` adds `minReleaseAge=3d` as a fallback, host deps are pinned (`@chat-adapter/telegram` `4.26.0`, `cron-parser` `5.5.0`, `better-sqlite3` `11.10.0`). (12) GitHub Actions: `.github/workflows/ci.yml` runs typecheck-host + typecheck-container + vitest + bun-test on every PR, plus `bump-version.yml`, `label-pr.yml`, `update-tokens.yml`. Significant CI surface, none of which exists on v1.

### Implementation comparison

- **Functionality:** Mount allowlist (validate user-supplied additional mounts against an external config)
- **v1 approach:** `src/mount-security.ts:232-343` accepts an `isMain: boolean` parameter, supports `nonMainReadOnly: true` to force RWтЖТRO downgrade for non-main groups (`mount-security.ts:310-319`), uses strict pattern matching `part === pattern || part.startsWith(pattern + '.') || part.startsWith('.' + pattern)` (`mount-security.ts:154-162`) тАФ comment explicitly cites avoiding the `my-credentials-app` false positive. Has `allowAbsoluteContainerPath` option for admin-installed plugin mounts (`mount-security.ts:223-226`). Backed by a 200+ line vitest suite.
- **v2 approach:** `src/modules/mount-security/index.ts:234-311` drops the `isMain` parameter entirely (no main-vs-non-main concept in v2), drops `nonMainReadOnly`, drops `allowAbsoluteContainerPath`. Adds `:` rejection in `isValidContainerPath` (`index.ts:215-217`) to defend against `-v <repo>:rw` injection. Pattern matching is `part === pattern || part.includes(pattern)` (`index.ts:152-170`) тАФ exactly the false-positive case v1's comment was warning about. Adds `secrets.json`, `token.json`, `.ssh-agent` to `DEFAULT_BLOCKED_PATTERNS` with a "Ported from nanotars v1" comment (`index.ts:57-61`). **No test file.**
- **Verdict:** Mixed. v2's `:` injection check is a real improvement (PORT to v1, trivial). v2's `part.includes(pattern)` pattern matching is a regression тАФ it will block legitimate paths like `~/projects/credentials-app/`. v1's strict-match is correct (KEEP). v2's lack of tests is a regression for a security-sensitive module. v1's `nonMainReadOnly` knob is irrelevant in v2 (no main concept) but still load-bearing in v1 (KEEP).

- **Functionality:** Outbound secret redaction
- **v1 approach:** `src/secret-redact.ts:53-170` builds a composite regex from `.env` + `groups/*/.env` + `~/.claude/.credentials.json`, NEVER_EXEMPT covers four critical secrets, no length-sort. Wired via `router.ts:77` (every outbound message) and `container-runner.ts:360-384` (every stdout/stderr log line written to disk).
- **v2 approach:** `src/modules/secret-redaction/index.ts:73-121` is a clean refactor: `LoadSecretsOptions` accepts injectable paths (testability), values are deduped via Set, sorted by length descending so longer secrets match before shorter prefixes (`index.ts:103`), and `ONECLI_API_KEY` is added to NEVER_EXEMPT. Wired via `delivery.ts:361` only тАФ there is no stdout/stderr redaction because v2 doesn't capture stdout (container writes to `outbound.db`). Comprehensive vitest suite (11 cases) at `secret-redaction.test.ts:1-112`.
- **Verdict:** PORT v2's redaction module to v1 тАФ better. The length-sort + Set-dedup + injectable paths + `ONECLI_API_KEY` exempt are all strict improvements. v1's stdout-redaction wiring on `container-runner.ts` must be kept (v1 still pipes stdout); only the module body should be replaced. Effort: small.

- **Functionality:** Auth-error detection
- **v1 approach:** `router.ts:7-21` тАФ `AUTH_ERROR_PATTERNS` array of seven case-insensitive substrings (`'does not have access to claude'`, `'oauth token has expired'`, `'obtain a new token'`, `'refresh your existing token'`, `'authentication_error'`, `'invalid_api_key'`, `'please login again'`). `isAuthError(text)` returns true if any pattern is a substring of the lowercased text. Consumed by `orchestrator.ts:272-283` (immediate user-facing "[Auth Error]" notice on the first auth-error chunk in a session) and `task-scheduler.ts:216-226` (scheduled-task error path adds `[Auth Error]` prefix and a fixed message instead of leaking the raw error).
- **v2 approach:** Absent. Confirmed by `grep -rn "isAuthError\|AUTH_ERROR\|oauth.*token.*expired\|invalid_api_key\|authentication_error" /data/nanoclaw-v2/src` returning zero hits, and `delivery.ts:33` showing the only error-related logic is `MAX_DELIVERY_ATTEMPTS = 3`.
- **Verdict:** CONTRIBUTE тАФ v1's `AUTH_ERROR_PATTERNS` + `isAuthError` is a clean two-function module v2 should adopt. The module is content-addressable (no v1-specific dependencies), and the call sites in v2 would be inside `delivery.ts` (replace silent attempt-bump with a one-shot user notification on auth signature) and inside any provider error path. Effort: small. Confidence: high. CONTRIBUTE because v1 has it, v2 doesn't.

- **Functionality:** Credential model
- **v1 approach:** `.env` is parsed by `src/env.ts:11-54` into a host-only Record<string,string> тАФ values are NOT loaded into `process.env` to keep them off child processes. The container-mounts builder (`container-mounts.ts:319-379`) then writes a filtered subset (allowlist from plugin manifests' `containerEnvVars`, default `['ANTHROPIC_API_KEY', 'ASSISTANT_NAME', 'CLAUDE_MODEL']`) to a per-group env file at `data/env/<groupFolder>/env`, single-quotes each value (`container-mounts.ts:371-378`), bind-mounts at `/workspace/env-dir` RO. Claude OAuth tokens are bind-mounted from `~/.claude/.credentials.json`. The agent has direct access to its credentials inside the container тАФ `docs/SECURITY.md:97-101` explicitly notes this and asks for PRs.
- **v2 approach:** OneCLI gateway. `src/container-runner.ts:486-500` calls `onecli.ensureAgent({ name, identifier: agentIdentifier })` then `onecli.applyContainerConfig(args, { addHostMapping: false, agent: agentIdentifier })` which injects `HTTPS_PROXY` + a CA cert mount into the run args. Every outbound HTTPS call from the container is rewritten to go through OneCLI's gateway, which injects the matching credential at request time per host pattern. Credentials never sit in the container env or filesystem. The agent group id is the OneCLI agent identifier тАФ credentials scope to agent groups, not channels. Per-group env still goes through `group-env/index.ts` for non-credential vars (NOTION_API_KEY-type things would still go via OneCLI, but config like `TZ` flows the env path). Approval-gating is a separate manual-approval flow (`onecli-approvals.ts`) that long-polls the gateway and DMs an admin.
- **Verdict:** ADOPT for v1 тАФ OneCLI is a separate daemon, distributable as a sidecar, and v1 explicitly admits the credential-leak problem. The `applyContainerConfig` SDK call is one line; the manual-approval handler is a self-contained module. The bigger lift is the operational surface (install, vault setup, agent registration), which is out of scope for the host code itself but documented in v2's `init-onecli` skill. Effort: medium-large. Confidence: medium тАФ depends on whether v1's plugin-loader env wiring can coexist with HTTPS_PROXY for credentialed calls (probably yes, since v2 keeps both).

- **Functionality:** IPC layer
- **v1 approach:** Filesystem JSON files under `data/ipc/<group>/{messages,tasks,input}/`. Host polls every `IPC_POLL_INTERVAL` ms (`ipc/index.ts:31-118`). Identity is the directory name; authorization is "main can do anything, non-main can only act on its own group's resources" (`ipc/auth.ts`). Every read uses `O_NOFOLLOW` + lstat + 1 MiB cap (`ipc/file-io.ts:16-40`). Quarantine on parse/type-guard failure (`ipc/file-io.ts:58-62`). Stdout protocol is a sentinel-bracketed JSON parsed by `output-parser.ts:34-89` тАФ `makeMarkers(nonce)` produces per-run sentinels so the agent can't inject a fake END marker and forge a payload. The IPC message types are a tight discriminated union (`ipc/types.ts:5-23`).
- **v2 approach:** Two SQLite DBs per session at `data/v2-sessions/<session_id>/{inbound,outbound}.db`. Host writes to `inbound.db.messages_in` and reads from `outbound.db.messages_out`; container does the inverse. Single-writer-per-file invariant. Heartbeat is a file touch (`/workspace/.heartbeat`). Even-seq host, odd-seq container. Identity is now the session тАФ there is no "group folder" naming anything. No stdout protocol, no JSON files, no `output-parser.ts`, no symlink/lstat checks (no untrusted disk paths in the message flow), no quarantine.
- **Verdict:** SKIP-ARCH. The two-DB session model is the architectural cutover; porting it to v1 would mean replacing the entire IPC layer, the per-group identity model, the orchestrator's stdout consumer, and the channel-router's reply path. That's ╬▓-territory тАФ out of scope for the upstream-triage work. v1's IPC has merits worth keeping in v1 (file-based debugging, simple scripting, quarantine for forensic recovery) and the migration story is `migrate-v2.sh`, not a port.

- **Functionality:** Bash security hooks
- **v1 approach:** `container/agent-runner/src/security-hooks.ts:17-134` тАФ two factory functions `createSanitizeBashHook()` and `createSecretPathBlockHook()`, returning `HookCallback` closures. Bash regex set: `/proc/*/environ`, `/tmp/input.json`, `.credentials.json`, plus `cat|less|head|tail|base64|xxd|strings|python|node|perl|ruby` paired with sensitive-path patterns. Read regex set: `/proc/*/environ`, `/tmp/input.json`, paths ending in `.credentials.json`. Prepends `unset ANTHROPIC_API_KEY; unset CLAUDE_CODE_OAUTH_TOKEN`. 21-test vitest suite.
- **v2 approach:** `container/agent-runner/src/security-hooks.ts:1-110` тАФ exported as constants, not factories (`sanitizeBashHook` and `secretPathBlockHook`). Regex set is consolidated and named (`PROC_ENVIRON_RE`, `INPUT_JSON_RE`, `CREDS_JSON_RE`, `READ_TOOLS_RE`, `SENSITIVE_PATH_RE`). `READ_TOOLS_RE` adds `more`, `od`, `hexdump`, `bun`, `awk`, `sed`, `python3`. `SECRET_ENV_VARS` adds `ONECLI_API_KEY`. 10-test bun:test suite. Functionally equivalent.
- **Verdict:** PORT тАФ v2's expanded `READ_TOOLS_RE` (`more|od|hexdump|bun|awk|sed|python3`) is a strict improvement. The factoryтЖТconstant change is cosmetic. v1 doesn't need `ONECLI_API_KEY` in `SECRET_ENV_VARS` unless v1 also adopts OneCLI. Effort: trivial.

- **Functionality:** Per-group env passthrough
- **v1 approach:** Allowlist comes from `pluginRegistry.getContainerEnvVars(channel, groupFolder)` (`container-mounts.ts:351-353`) тАФ plugin manifests declare `containerEnvVars: string[]`, the registry merges per-channel and per-group declarations, with a baked-in default of `['ANTHROPIC_API_KEY', 'ASSISTANT_NAME', 'CLAUDE_MODEL']` if no plugin registry is wired. Values are single-quote-escaped (`container-mounts.ts:371-378`). `CLAUDE_MODEL` can be overridden by `store/claude-model` and per-task `modelOverride`. Per-group `.env` overlays global `.env`.
- **v2 approach:** `src/modules/group-env/index.ts:104-137` тАФ allowlist is `containerConfig.envAllowlist: string[]` from per-group `container.json` (no plugin registry). `parseEnvFile` (`index.ts:58-84`) reads each `.env`, group overrides global, `shellQuote` (`index.ts:94-96`) wraps in single quotes with `'` тЖТ `'\''` escape. If allowlist is empty, function returns null and no env mount is created (opt-in default). Has `group-env.test.ts` covering shellQuote and parseEnvFile and buildGroupEnvMount.
- **Verdict:** Neutral architecturally тАФ v1 sources allowlist from plugins (matches its plugin model), v2 sources it from `container.json:envAllowlist` (matches its agent-group model). v2's module is cleaner code (injectable paths, explicit factor-out, unit tests) but the data flow is the same shape. Earlier scope notes claimed "v1 merges all of `.env`" тАФ that's wrong, v1 has its own filter. **PORT** v2's `shellQuote` test cases to v1's container-mounts test (it has no shellQuote unit tests today). Effort: trivial.

- **Functionality:** Container hardening flags
- **v1 approach:** `src/container-runtime.ts:155-177` adds `--cap-drop=ALL`, `--cap-add=SYS_PTRACE`, `--security-opt=no-new-privileges`, `--security-opt seccomp=<chromium-seccomp.json>`, `--shm-size=2g`, `--init`, `--cpus=2`, `--memory=4g`, `--pids-limit=256`, `--add-host=host.docker.internal:host-gateway` for every Docker container spawn. Apple Container path returns no extra args.
- **v2 approach:** `src/container-runner.ts:473-548` runs with only `--rm`, `--name`, `--label`, optional `--user <uid>:<gid>`, `--entrypoint bash`, plus volume mounts, env vars, and OneCLI's `applyContainerConfig` injection. **No cap-drop, no cap-add, no no-new-privileges, no seccomp profile, no resource limits, no PID limit.** `tini` remains in the image as PID 1 (`container/Dockerfile:121`) for signal forwarding only. `grep -rn "cap-drop\|cap-add\|no-new-privileges\|seccomp\|--shm-size\|--cpus\|--memory\|--pids-limit"` across `src/`, `container/`, `setup/` returns no hits.
- **Verdict:** CONTRIBUTE / KEEP тАФ v1's hardening posture is strictly better for any deployment with non-trusted code paths reaching the container. v2's regression is significant: a compromised agent-runner has full container caps, can spawn unbounded processes, and can set-uid-on-exec. KEEP v1's hardening; CONTRIBUTE the entire `extraRunArgs()` block back to v2 along with the `chromium-seccomp.json` file. Effort: trivial (ship the same flags). Confidence: high. Note: `--memory=4g` and `--cpus=2` may need tuning per v2's per-session-container model (since v2 spawns one container per session, not per group).

- **Functionality:** Logging
- **v1 approach:** `pino` + `pino-pretty` (`logger.ts:1-16`). Structured JSON in production (`pino-pretty` only with NODE_ENV=development implicitly via the import), uncaught + unhandled rejection trapping, `LOG_LEVEL` env var.
- **v2 approach:** Hand-rolled `src/log.ts:1-65`, ANSI-coloured, no pino. Same `LOG_LEVEL` semantics, same uncaught/unhandled traps, different API shape (`log.info(msg, data?)` instead of `logger.info({ data }, msg)`). No structured JSON output mode; not pluggable.
- **Verdict:** KEEP v1's pino. v2 has dropped a real dependency for hand-rolled formatting; pino offers structured logs, log shipping integrations, child loggers, and redaction primitives that v2's hand-roll doesn't. v1's wiring is one of the few places it's strictly better than v2. Don't PORT v2's log.ts to v1 тАФ it's a regression on the dimension. Effort: zero (no action).

- **Functionality:** Tests + CI

Effort: small (copy `ci.yml`, add npm-test command). Separately, **CONTRIBUTE the v1 mount-security test back to v2** тАФ the test exists and is largely portable to vitest with v2's module-shape adjustments. Effort: small.

The pnpm migration is a real change but possible to do incrementally. Effort: medium. Confidence: high. - **Functionality:** Operational skills inventory

### What surprised me

1. **v2 dropped every container hardening flag**. v1's `container-runtime.ts:155-177` ships nine non-trivial Docker flags тАФ `--cap-drop=ALL`, `--cap-add=SYS_PTRACE`, `--security-opt=no-new-privileges`, custom seccomp profile, `--shm-size=2g`, `--init`, `--cpus=2`, `--memory=4g`, `--pids-limit=256`. v2's `container-runner.ts:473-548` ships exactly zero of them. `tini` is in the image as PID 1, but that's signal forwarding, not security. This is the most consequential security regression in this area: a compromised agent in v2 has full container capabilities, can fork-bomb, can `sudo` if any setuid binaries exist. The migration story emphasises ╬▓-posture for architectural ports, but this isn't architectural тАФ it's nine flags on a `docker run` invocation. Either v2 has a reason for dropping them I'm missing, or this is a clear CONTRIBUTE.

2. **v2 has no auth-error detection at all**. I went in expecting v2 to have a fancier replacement using OneCLI's typed errors or the Agent SDK's structured error events, since the credential model is so different. But `delivery.ts:33` shows the entire error path is `MAX_DELIVERY_ATTEMPTS = 3` тАФ three retries and a silent give-up. v1's seven-pattern substring match is crude, but it gives the user a deterministic "[Auth Error]" message instead of a session that just stops responding. The regression here is silent rather than loud; users on v2 will probably just notice their bots stop replying after a token expires.

3. **v2's mount-security port has a real regression in pattern matching**. v2's `matchesBlockedPattern` (`index.ts:152-170`) uses `part.includes(pattern)`, which means a path like `~/projects/credentials-app/` matches the `credentials` blocked pattern. v1's `mount-security.ts:154-162` deliberately uses strict matching with a code comment citing exactly this false-positive case тАФ `// substring matching caused false positives (e.g. blocking "my-credentials-app" because it contains "credentials")`. v2 either silently re-introduced the bug v1 fixed, or made a deliberate correctness tradeoff that's not documented. With no v2 test file to encode the intent, it's impossible to tell which.

### Cross-cutting concerns

- **Auth-error detection тЖФ Persistence layer (Area 1) тЖФ Runtime (Area 3).** Adding `isAuthError` to v2 needs a host-side hook in `delivery.ts` (Area 4 channels) for live messages and a hook in the scheduling tick (Area 3) for scheduled-task error reporting. The error text plumbing flows through the per-session DB (Area 1's `messages_out` table), which is where we'd read the candidate text. Listed under both Area 1 (text comes from DB) and Area 3 (tick logic).
- **OneCLI credential model тЖФ Permissions/approvals (Area 2).** The OneCLI manual-approval bridge depends on Area 2's `pickApprover` + `pickApprovalDelivery` primitives and on the `pending_approvals` table schema. Adopting OneCLI without the approvals primitive would mean re-implementing the approver-selection logic.
- **Container hardening тЖФ Runtime (Area 3).** The flags live in `container-runtime.ts`, which Area 3 owns. If Area 3 keeps v1's container-runtime intact (likely under SKIP-ARCH for the runner itself), the hardening flags come along for free. If Area 3 rebuilds container-runtime to look more like v2's, the hardening flags need to be carried over explicitly.
- **Mount-allowlist tests тЖФ none тАФ orphan port.** v1's `mount-security.test.ts` can be portably contributed back to v2 as a standalone PR; no other area depends.
- **Secret redaction module тЖФ Channels (Area 4).** v2 only redacts on the host-side delivery path; v1 also redacts at stdout-write. Porting v2's redaction body to v1 must preserve v1's two redact call-sites (router.ts and container-runner.ts). v2's wiring isn't portable without v2's two-DB model.
- **`group-env` module тЖФ Extensions (Area 5).** v1's env allowlist is sourced from plugin manifests (`plugin-loader.ts:43-44, 69, 264`). v2's env allowlist is per-group `container.json`. The two source-of-truth models conflict тАФ porting v2's module body to v1 means deciding whether to keep the plugin-registry source (Area 5's territory) or replace it with `container.json:envAllowlist`. This is bigger than the body of group-env itself.
- **CI тЖФ Tests.** Adopting v2's `ci.yml` requires v1's vitest suite to be green on `ubuntu-latest`. Worth a one-shot manual run to confirm before adding the workflow.
- **Logging тЖФ everything.** `log.ts` (v2) vs `logger.ts` (v1) are not interchangeable APIs (`log.info(msg, data?)` vs `logger.info({ data }, msg)`). Any code ported across has to account for the surface change. Recommendation: KEEP v1's pino, rewrite v2-imported code's logger calls during port тАФ small, mechanical.

### Open questions

1. **Why did v2 drop container hardening?** Was this a deliberate tradeoff (e.g. OneCLI requires capabilities the seccomp profile blocks) or an accidental regression during the v2 cutover? Need to spot-check a v2 commit that touched `container-runtime.ts` or `container-runner.ts` around the 2.0.0 bump (`fe942dd`). If accidental, this becomes a high-priority CONTRIBUTE; if deliberate, the rationale needs documenting before any port is attempted.

2. **Is the mount-security `part.includes` regression intentional?** v2's commit history will show whether the change from `===` to `includes` was a refactor mistake or a deliberate widening. Without tests, the intent isn't recoverable from the source. If it's a regression, we can CONTRIBUTE v1's stricter match back; if it's intentional, we need to understand the threat model that motivated the change.

3. **Can the OneCLI approval flow be retrofitted to v1 without the `pending_approvals` schema?** v1 has no equivalent table тАФ its closest analogue is the IPC tasks for emergency stop. Adopting OneCLI would either require introducing the `pending_approvals` table (Area 2 territory) or hand-rolling a simpler in-memory pending map. Needs a design decision before estimating effort.

4. **Does v1's pino-pretty work in production?** `logger.ts:5` declares `transport: { target: 'pino-pretty' }` unconditionally тАФ that's a production dep, not a dev-only one. If the pretty transport is doing stdio formatting at runtime, that's potentially a perf hit relative to plain JSON. Should v1's logger be configured to fall back to plain pino in production? Low priority but worth flagging.

5. **Is v2's missing `mount-security` test file a known gap or an oversight?** If the v2 maintainers know the test coverage is missing and they're working on it, the v1-test-CONTRIBUTE may be redundant. If they don't know, the contribution is high-value. Easy to confirm with one issue/grep.
