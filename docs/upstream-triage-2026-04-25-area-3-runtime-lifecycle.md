## Area: Runtime, lifecycle, scheduling

### Functional inventory

- **v1-archive (nanotars):** A single host process orchestrates **per-agent-group** containers via `runContainerAgent` (`/data/nanotars/src/container-runner.ts:103-461`), serialised through a `GroupQueue` (`/data/nanotars/src/group-queue.ts:31-425`) that enforces both a global concurrency cap (`MAX_CONCURRENT_CONTAINERS`, default 5 — `/data/nanotars/src/config.ts:60-62`) and per-group at-most-one-active. The container is a long-lived `docker run -i --rm` invocation: the host pipes a JSON `ContainerInput` (with secrets) on stdin, streams stdout through `output-parser.ts` markers, and follow-up messages reach the live container by file IPC under `data/ipc/<group>/input/` (`group-queue.ts:187-220`). The container runtime is auto-detected (`/data/nanotars/src/container-runtime.ts:21-42`) — Docker first, Apple Container fallback — and Docker spawns get a hardened arg block: `--cap-drop=ALL`, `--cap-add=SYS_PTRACE`, `--security-opt=no-new-privileges`, custom Playwright seccomp, `--shm-size=2g`, `--init`, `--cpus=2`, `--memory=4g`, `--pids-limit=256`, `--add-host=host.docker.internal:host-gateway` (`container-runtime.ts:156-177`). The container Dockerfile (`/data/nanotars/container/Dockerfile`) is `node:22-slim`, installs `chromium`, `agent-browser`, and `@anthropic-ai/claude-code` globally via `npm install -g`, runs as `USER node` (UID 1000), and uses `tsx src/index.ts` as entrypoint (no tini — `--init` covers reaping). Scheduling is a polled loop (`/data/nanotars/src/task-scheduler.ts`): `getDueTasks()` every `SCHEDULER_POLL_INTERVAL = 60_000` ms (`config.ts:16`), `claimTask` clears `next_run` to prevent re-enqueue, then `enqueueTask` routes through the same `GroupQueue` (with idle preemption — `group-queue.ts:122-136`). Tasks store their schedule as `(schedule_type, schedule_value)` where `schedule_type ∈ {cron, interval, once}` (`/data/nanotars/src/types.ts:74-76`); recurrence is computed in-process via `cron-parser` (`task-scheduler.ts:43-62`). Scheduled-task failures are notified back to the chat ("[Scheduled task failed] …" or "[Auth Error] …" — `task-scheduler.ts:212-227`). Per-task model overrides flow as `task.model` and become an env var via `buildVolumeMounts` (`/data/nanotars/src/container-mounts.ts:362-368`). Pause/resume is a separate emergency-stop path: `GroupQueue.emergencyStop` (`group-queue.ts:359-383`) plus `MessageOrchestrator.pause/resume` (`/data/nanotars/src/orchestrator.ts:175-201`).

- **upstream/main (nanoclaw v2):** A single host process spawns **per-session** containers (one per `(agent_group, messaging_group, thread_id)` triple) via `wakeContainer` → `spawnContainer` (`/data/nanoclaw-v2/src/container-runner.ts:79-185`). There is no queue: each session has its own container, mounted on its own `data/v2-sessions/<session_id>/` directory containing `inbound.db`, `outbound.db`, `outbox/`, and `.claude/`. All host↔container IO is through those two SQLite files; container stdin is `'ignore'` (`container-runner.ts:152`) and stdout is unused. Concurrency is gated only by an in-flight wake dedupe map (`wakePromises` — `container-runner.ts:63-97`); the host-wide cap exists in config (`/data/nanoclaw-v2/src/config.ts:40` — `MAX_CONCURRENT_CONTAINERS = 5`) but is **never read anywhere** in the runtime code path. The container runtime is hard-coded to Docker (`/data/nanoclaw-v2/src/container-runtime.ts:12` — `CONTAINER_RUNTIME_BIN = 'docker'`); Apple Container fallback was dropped. Docker spawn args are minimal (`container-runner.ts:464-549`): `--rm`, `--name`, `--label`, `-e TZ`, OneCLI gateway via `onecli.applyContainerConfig`, `host.docker.internal` add-host on Linux, optional `--user` for non-1000 UIDs, RO/RW mounts, `--entrypoint bash` plus `bun run /app/src/index.ts`. **No `--cap-drop`, no `--security-opt=no-new-privileges`, no `--shm-size`, no `--cpus`, no `--memory`, no `--pids-limit`, no seccomp profile.** The Dockerfile (`/data/nanoclaw-v2/container/Dockerfile`) is `node:22-slim`, uses pinned `ARG`s for claude-code/agent-browser/vercel/bun versions, installs `tini` and uses it as PID-1 entrypoint (`Dockerfile:121` — `ENTRYPOINT ["/usr/bin/tini", "--", "/app/entrypoint.sh"]`), pnpm-installs CLIs via `corepack` with explicit `only-built-dependencies` allowlist for `agent-browser` + `@anthropic-ai/claude-code`, and bakes Bun in via the official installer pinned to `BUN_VERSION`. Source is **not baked in**: `/app/src` is a host-side RO bind mount (`container-runner.ts:340-341`). Stuck/idle detection is delegated entirely to `host-sweep.ts` (`/data/nanoclaw-v2/src/host-sweep.ts`) which runs every 60s, computes `decideStuckAction` against heartbeat-file mtime, processing-claim age, and per-tool declared timeout (`host-sweep.ts:71-107`), and kills via `killContainer` for the absolute ceiling (default 30 min, extended for declared Bash timeouts) or per-claim stuck (default 60s, also Bash-extended). Pause/resume is a small lifecycle module (`/data/nanoclaw-v2/src/modules/lifecycle/index.ts`) gated by a process-level `paused` flag that `wakeContainer` checks (`container-runner.ts:80-83`); it bulk-kills via `killAllContainers` and is fired by agent-emitted `emergency_stop` / `resume_processing` system actions (`/data/nanoclaw-v2/src/modules/lifecycle/actions.ts`). v2 adds **self-modification** (`/data/nanoclaw-v2/src/modules/self-mod/`): two delivery actions `install_packages` and `add_mcp_server` queue admin approvals (`request.ts:30-91`); on approve, `apply.ts:21-66` updates `container.json`, calls `buildAgentGroupImage` to build a per-agent-group image (`container-runner.ts:610-659`, layered Dockerfile in `generateAgentGroupDockerfile:569-607`), and kills the container so the next wake respawns on the new image. v2 adds a **provider abstraction**: host-side `provider-container-registry.ts` (`/data/nanoclaw-v2/src/providers/provider-container-registry.ts`) lets a provider self-register host mounts and env vars (resolved precedence: `sessions.agent_provider → agent_groups.agent_provider → container.json.provider → 'claude'` — `container-runner.ts:226-232`); container-side `providers/factory.ts` + `provider-registry.ts` resolve the provider implementation that wraps the SDK call (`/data/nanoclaw-v2/container/agent-runner/src/providers/claude.ts:240-345` + `mock.ts`). Scheduling is module-bound (`/data/nanoclaw-v2/src/modules/scheduling/`): tasks are stored as `messages_in` rows with `kind='task'` (no separate table — `db.ts:7-12`); `process_after` doubles as `next_run`; recurrence is the `cron-parser` expression on the row itself; `handleRecurrence` runs each sweep tick and inserts a fresh row with the next `process_after` (`recurrence.ts:21-54`); per-task model overrides ride in the JSON `content.model` and are extracted batch-side in the agent-runner (`poll-loop.ts:460-480`); critically, scheduling adds a **pre-task script hook** (`/data/nanoclaw-v2/container/agent-runner/src/scheduling/task-script.ts`) — a 30-second bash script can run before the agent and gate task execution via `wakeAgent: bool` JSON output, with the result fed back into the prompt as `scriptOutput`.

### Implementation comparison

#### Concurrency / queueing model

- **v1 approach:** `GroupQueue` (`group-queue.ts:31-425`) maintains a per-group `GroupState` plus a global `activeCount`. `enqueueMessageCheck` (line 78) and `enqueueTask` (line 104) check both per-group active-flag and `activeCount >= MAX_CONCURRENT_CONTAINERS` before spawning. Excess work parks in `pendingMessages`/`pendingTasks` and a `waitingGroups` set; `releaseSlot` (line 273) → `drainGroup` (line 309) → `drainWaiting` (line 335) walks the wait queue when a slot frees. Tasks preempt idle containers: `notifyIdle` (line 162) writes a `_close` sentinel if pending tasks exist (line 168). Retry uses exponential backoff capped at `MAX_RETRIES = 5` (line 286-307). One container per group at a time: a chat conversation and a scheduled task for the *same* group serialise through the same container.
- **v2 approach:** No queue at all. `wakeContainer` (`container-runner.ts:79-98`) returns immediately if a container is already running for that session id; otherwise it spawns. The dedupe is *per-session*, not per-agent-group — two sessions for the same agent group on different threads spawn two parallel containers. `MAX_CONCURRENT_CONTAINERS` is exported from `config.ts:40` but is never imported anywhere outside its own definition (`grep` confirms). Stuck-detection happens in `host-sweep.ts` on a 60s schedule, not synchronously around spawn. Retry/backoff for stale `processing` rows happens in `resetStuckProcessingRows` (`host-sweep.ts:251-295`) with the same `MAX_TRIES = 5` + `BACKOFF_BASE_MS = 5000` constants v1 uses — but applied per *message* row, not per group.
- **Verdict:** Mixed. v2's per-session model is architecturally tied to the two-DB rewrite and gives clean per-thread isolation, but losing the host-wide cap is **a real regression** for resource-constrained hosts: a burst of inbound on N sessions = N concurrent Chromium-capable containers regardless of CPU/RAM budget. The presence of dead `MAX_CONCURRENT_CONTAINERS` config strongly suggests the cap was originally intended and got dropped during the rewrite. v1's queue is also more complex than necessary for nanotars's per-group model — combining the two: keep v1's queue, optionally lift v2's session granularity to a future enhancement.

#### Container hardening flags

- **v1 approach:** Docker spawns get the full hardening arg block in one place — `extraRunArgs()` at `container-runtime.ts:156-177`. `--cap-drop=ALL --cap-add=SYS_PTRACE --security-opt=no-new-privileges --security-opt seccomp=container/chromium-seccomp.json --shm-size=2g --init --cpus=2 --memory=4g --pids-limit=256`. Apple Container path returns `[]` because Apple's runtime applies its own VM-level isolation. Permissions are reconciled by `fixMountPermissions` (`container-runtime.ts:192-202`) which `chown -R 1000:1000`s every writable mount on Docker.
- **v2 approach:** None of the cap-drop / security-opt / resource-limit / seccomp flags are applied. `buildContainerArgs` (`container-runner.ts:464-549`) goes straight from `--rm --name --label` to env, OneCLI args, host gateway, optional `--user`, mounts, entrypoint. `tini` inside the image gives proper signal forwarding (Dockerfile line 121), and the image runs as `USER node` (Dockerfile line 116), but the kernel-level container hardening v1 has is gone. Apple Container support is gone too — `CONTAINER_RUNTIME_BIN` is a hard-coded `'docker'` string (`container-runtime.ts:12`).
- **Verdict:** v1 is materially better here. The hardening block matters: a Chromium-driving agent in an unhardened container is a real escape risk surface, and `--memory` + `--pids-limit` cap blast radius for a runaway agent. This is a clean PORT (the args are platform-agnostic) and likely worth a CONTRIBUTE PR upstream.

#### Container runtime abstraction

- **v1 approach:** `container-runtime.ts:21-42` does runtime detection at first call (Docker → Apple Container fallback), caches the choice, and exposes `cli()`, `run()`, `extraRunArgs()`, `stop()`, `fixMountPermissions()`, `ensureRunning()`, and orphan cleanup that handles both runtimes' container-listing formats (`container-runtime.ts:85-148`).
- **v2 approach:** Hard-coded Docker (`container-runtime.ts:12`). `ensureContainerRuntimeRunning()` runs `docker info` only. Orphan cleanup uses a label filter (`label=nanoclaw-install=<slug>`) which lets multiple installs share a host without reaping each other's containers (`container-runtime.ts:67-90`) — this is genuinely better than v1's name-prefix matching.
- **Verdict:** Mixed. v2's label-scoped orphan cleanup is a nicer model and worth keeping; v1's runtime abstraction is a larger interface that v2 chose to give up. For nanotars, KEEP v1's abstraction (Apple Container support is in active use). The label-scoping approach from v2 is a small ADOPT — replace v1's name-prefix `'nanoclaw-'` filter with a per-install slug.

#### Idle / timeout / signal handling

- **v1 approach:** Two timers per spawn — a hard `timeoutMs = max(configTimeout, IDLE_TIMEOUT + 30s)` (`container-runner.ts:188-191`) and an idle-stdin closer in `orchestrator.processGroupMessages` (`orchestrator.ts:248-254`) that writes `_close` after `IDLE_TIMEOUT = 1_800_000` ms (`config.ts:52-54`). Scheduled-task variant uses `SCHEDULED_TASK_IDLE_TIMEOUT = 30_000` ms (`config.ts:56-58`). On hard timeout it calls `containerRuntime.stop` first, falls back to `SIGKILL`. Signal forwarding into the container is via Docker's `--init` (zombie reaper / signal proxy).
- **v2 approach:** No host-side wall-clock timer on spawn. Stuck detection is decoupled to the 60s sweep + heartbeat-file mtime. Inside the container, the agent-runner calls `touchHeartbeat()` on every SDK event (`poll-loop.ts:301`, `task-script.ts:104,106`). The host's `decideStuckAction` widens its tolerance dynamically: if Bash declared a long timeout (read from `container_state.tool_declared_timeout_ms` written by the `preToolUseHook` at `claude.ts:151-170`), the ceiling becomes `max(30 min, declared_bash_ms)` (`host-sweep.ts:71-94`). Signal forwarding is via `tini` as PID 1 (Dockerfile:121), which is the cleanest available option.
- **Verdict:** v2's heartbeat-driven model is better in concept — it doesn't kill legitimately-long-running work, doesn't need a separate "scheduled task" timeout, and the per-tool widening is elegant. But it depends on the two-DB architecture (`outbound.db.container_state`, heartbeat file at `/workspace/.heartbeat`) and is SKIP-ARCH for nanotars on its own. For v1 the existing dual-timer is fine; the heartbeat idea isn't worth porting without the rest of the rewrite.

#### Pause / resume

- **v1 approach:** `GroupQueue.emergencyStop` (`group-queue.ts:359-383`) sets `shuttingDown = true`, races a 10s `Promise.all(stop)` against all active containers. `MessageOrchestrator.pause/resume` (`orchestrator.ts:175-201`) uses `stopRequested` plus a wake event to suspend/restart the message loop. Two separate paths — queue + loop — must both be paused to fully silence the host.
- **v2 approach:** Single module (`modules/lifecycle/`), 60 lines total. `pause()` flips a process flag, calls `killAllContainers` (one bulk path), logs and returns the killed list (`index.ts:30-50`). The flag gates `wakeContainer` (`container-runner.ts:80-83`) so future inbound just queues in `inbound.db` until resume. Intentional: outbound polls keep draining already-produced messages so in-flight work still reaches the user. Hookup via `registerDeliveryAction` for `emergency_stop` / `resume_processing` (`module.ts:11-12`); any agent can fire either action — no role check (`actions.ts:11-13` notes this is fine for single-operator installs).
- **Verdict:** v2 is materially simpler and the agent-emitted system-action surface is nicer than v1's bespoke pause/resume. But it depends on two-DB + delivery-action registry + per-session containers. SKIP-ARCH for nanotars-as-v1; the simplification only buys you the cleaner shape if you've already done the wider rewrite.

#### Self-modification (install_packages / add_mcp_server)

- **v1 approach:** None. Any agent-driven container mutation requires an admin manually editing config + rebuilding the image.
- **v2 approach:** Two-action flow (`/data/nanoclaw-v2/src/modules/self-mod/`). Container's MCP tools write a `kind='system'` outbound message with `action: install_packages` (apt + npm arrays + reason) or `action: add_mcp_server` (name, command, args, env). Delivery dispatches to `request.ts` (`request.ts:20-91`), which sanitises (apt: `^[a-z0-9][a-z0-9._+-]*$`; npm: optional scope `@x/`; ≤ 20 packages — line 31-53), then `requestApproval` queues an admin DM. On approve, `apply.ts:21-66` updates `container.json`, runs `buildAgentGroupImage` (`container-runner.ts:610-659`) which writes a per-group Dockerfile (`generateAgentGroupDockerfile:569-607` — `FROM <base>` → `USER root` → optional apt + pnpm install with allowlist + dockerfile partials → `USER node`), tags it `nanoclaw-agent:<agentGroupId>`, stores the tag in `containerConfig.imageTag`, kills the container, and schedules a 5s-deferred follow-up message asking the agent to verify. `add_mcp_server` skips the rebuild and just kills (bun runs TS directly so a config change is enough).
- **Verdict:** v2-only feature, depends on two-DB + approval module + per-group `container.json`. SKIP-ARCH for nanotars under (β); the underlying idea (admin-approved container mutation) is worth revisiting under (γ) but isn't portable as-is. Note the approval-routing piece is reusable in principle, but it lives on top of the v2 user_roles model.

#### Provider abstraction

- **v1 approach:** Hard-wired Anthropic SDK in `container/agent-runner/src/index.ts:567-606` — single `query({prompt, options:{...}})` call with `mcpServers` inline; model is set via `process.env.CLAUDE_MODEL`; no provider seam. Adding OpenCode/Codex/Ollama would be a fork of the file.
- **v2 approach:** Two-sided. (1) Container-side: `AgentProvider` interface with `query(input)`, `isSessionInvalid(err)`, `supportsNativeSlashCommands` (`/data/nanoclaw-v2/container/agent-runner/src/providers/types.ts`). Each provider self-registers via `registerProvider('name', factory)` (`claude.ts:347`, `mock.ts:77`). The `factory.ts` resolves at startup from `container.json.provider`. The poll-loop is provider-agnostic — it formats the prompt once, calls `provider.query()`, drains `provider.events` looking for `init`/`result`/`error`/`progress`/`activity` (`poll-loop.ts:170-216`, `claude.ts:299-333`). (2) Host-side: `provider-container-registry.ts` lets a provider declare extra mounts/env that the host should add at spawn (`provider-container-registry.ts:34-50`); `resolveProviderName` walks `session → group → container.json → 'claude'` (`container-runner.ts:226-232`), and `resolveProviderContribution` invokes the registered host-side fn (`container-runner.ts:234-249`). Trunk ships only the `claude` provider plus a `mock`; OpenCode/Codex/Ollama are skill-installable per `add-opencode` / `add-codex` / `add-ollama-provider`.
- **Verdict:** v2-only. Cleanly designed but depends on `container.json` (per-group) plus `agent_provider` columns on `agent_groups` + `sessions` + per-provider XDG/env handling. SKIP-ARCH for nanotars under (β); the seam itself is small, but the value only kicks in when you actually want non-Claude providers, which is wider scope than runtime triage. Worth re-scoping if/when γ or specifically for adding Ollama.

#### Scheduled tasks — storage shape

- **v1 approach:** Dedicated `scheduled_tasks` table (`/data/nanotars/src/db/init.ts:46-52`) with `(schedule_type, schedule_value, next_run, status, last_run, last_result, context_mode, model)`. Separate `task_run_logs` table for history (`init.ts:60-68`). Recurrence is computed and re-stored via `updateTaskAfterRun` setting the next `next_run` (`tasks.ts:119-132`). `claimTask` (`tasks.ts:102-104`) clears `next_run` to take a row out of `getDueTasks` rotation while running.
- **v2 approach:** No dedicated table — tasks are `messages_in` rows with `kind='task'` (`scheduling/db.ts:7-12`); recurrence is a cron string on the row itself; `process_after` doubles as next-run; `series_id` ties recurrence chain together (`db.ts:30-31`); pause/cancel match by `(id OR series_id)` to hit the live next occurrence rather than just the completed row. No `task_run_logs` equivalent — message status is the audit trail.
- **Verdict:** Different shapes, both internally consistent. v1's separate table makes per-task history queries cheap; v2's row-as-task piggybacks on the inbound queue and gets retry/claim semantics for free. SKIP-ARCH the schema change; the storage choice is downstream of the two-DB design. v1's run-logs table is worth KEEPing even with v2's other ideas (audit/UI is useful).

#### Scheduled tasks — cron parser, recurrence handling

- **v1 approach:** `computeNextRun` (`task-scheduler.ts:43-62`) handles three schedule types — cron (CronExpressionParser with `tz: TIMEZONE`), interval (anchors to `next_run + ms`, skips missed intervals to prevent drift), and `once` (returns null). Result lands in `updateTaskAfterRun` after the run.
- **v2 approach:** Only cron supported. `handleRecurrence` (`recurrence.ts:21-54`) imports `cron-parser`, calls `interval.next()`, inserts a fresh `messages_in` row with `process_after = nextRun`, clears recurrence on the original. No `interval` or `once` types. Both apply `tz: TIMEZONE` (v2 file even cites v1 as the source of the TZ behaviour, recurrence.ts:30-31).
- **Verdict:** v1 is richer (interval + once support). The v2 author explicitly cited the v1 TZ behaviour; v2 dropped interval/once during the schema rewrite. KEEP v1's three-type model — it's already shipping and useful.

#### Scheduled tasks — atomic claim, idle preemption, error notification

- **v1 approach:** `claimTask` clears `next_run` *before* spawn (`task-scheduler.ts:84`), preventing re-enqueue while running. `notifyIdle`/preemption: an active container marked idle gets `_close` written if a task arrives (`group-queue.ts:122-136`, `162-171`) so tasks don't wait for full idle timeout. Error notification: `task-scheduler.ts:212-227` distinguishes auth errors (`isAuthError`) from generic failures and DMs the chat the task was scheduled from (`[Auth Error] …` vs `[Scheduled task failed] …`).
- **v2 approach:** Atomic claim via `markProcessing` setting status='processing' transactionally before the agent call (`poll-loop.ts:94`). No idle preemption needed — each session is its own container. Task-failure notification is on host-sweep `notifyTaskFailure` (`host-sweep.ts:306-344`) — only fires after `MAX_TRIES` retries, only for `kind='task'` rows, parses `content.prompt` to derive the task label, sends via the live delivery adapter (so it works even when the container is dead). v2 cites v1 as the inspiration: "Ported from nanotars v1 task-scheduler error-notification behaviour" (host-sweep.ts:278-279).
- **Verdict:** Both atomic-claim implementations are correct. v2's notification-only-after-MAX_TRIES is more conservative; v1 notifies on first failure (which is sometimes annoyingly noisy for transient errors but valuable for auth issues). v1's `isAuthError` discrimination is stronger; v2 doesn't differentiate auth from generic errors in the task-failure notify path. CONTRIBUTE: the auth-vs-generic split.

#### Scheduled tasks — pre-task script hook (v2-only)

- **v1 approach:** None. A scheduled task is "wake the agent unconditionally and run the prompt".
- **v2 approach:** Tasks can carry a `script` field (bash, ≤ 30s, ≤ 1 MB stdout — `task-script.ts:7-8`). Container runs the script before invoking the agent (`poll-loop.ts:141-150`); script's last stdout line must be JSON `{wakeAgent: bool, data?: any}`. `wakeAgent: false` → task row is marked completed without the agent ever running (e.g. "no new emails, skip"). `wakeAgent: true` → script's `data` is glued into the prompt as `scriptOutput` so the formatter can inject it. Script runs in the container's bash with the container's env (sees the agent's tools allowlist + secrets). `applyPreTaskScripts` is invoked via `MODULE-HOOK:scheduling-pre-task` (`poll-loop.ts:141-150`) — the marker block is empty when the scheduling module isn't installed, so no-op fallback is clean.
- **Verdict:** v2-only feature, **explicitly called out for investigation** in the spec. Genuine new capability, not derivable from v1. Cheap pre-checks (curl/grep/exit codes) avoid burning model tokens on no-op runs. Depends on `messages_in.content` being JSON (v1's `scheduled_tasks` is flat columns) and on the v2 module-hook pattern. Architecturally: porting requires extending `scheduled_tasks` schema (`script TEXT NULL`) plus wiring a pre-call exec in `task-scheduler.ts:142-175` before `runContainerAgent`. Self-contained, mostly orthogonal to the rest of v2's rewrite. ADOPT for nanotars at small/medium effort.

#### Scheduled tasks — per-task model override

- **v1 approach:** `task.model` column on `scheduled_tasks` (default `'claude-sonnet-4-5'` — `tasks.ts:20`). `task-scheduler.ts:153` passes it as `input.model` to `runContainerAgent`, which threads it into `buildVolumeMounts` → env file (`container-mounts.ts:362-368`) so the SDK reads `CLAUDE_MODEL`.
- **v2 approach:** Stored in the task row's content JSON as `content.model` (`actions.ts:31-43`). Validated host-side ("must start with `claude-`" — `actions.ts:32-33`). Read in the agent-runner via `extractTaskModelOverride` (`poll-loop.ts:460-480`) which also handles mixed-model batches by warning + using the first. Provider-agnostic at the abstraction layer (passed as `modelOverride` in `QueryInput` — `types.ts:60-66`); ClaudeProvider injects it as the `model` SDK option (`claude.ts:283`).
- **Verdict:** Both work; v2 is slightly nicer because the override travels with the message rather than as an out-of-band env mutation. But the storage shape is tied to `messages_in.content` being JSON. KEEP v1's model column; the env-file approach is fine for v1's architecture.

#### Container Dockerfile + build

- **v1 approach:** `node:22-slim` + global `npm install -g agent-browser @anthropic-ai/claude-code` (no version pins — `Dockerfile:44`). `tsx` for runtime TS execution; image *does* bake source via `COPY container/agent-runner/`, but also recompiles at startup. Plugin Dockerfile partials are concatenated by `build.sh` between the base and `USER node` line (`build.sh:33-60`).
- **v2 approach:** `node:22-slim` + pinned `ARG`s for every CLI version (`Dockerfile:23-25`), `tini` PID 1 (line 121), Bun runtime installed via official script pinned to `BUN_VERSION`, pnpm via corepack with explicit `only-built-dependencies` allowlist for postinstall scripts (line 96-105), `INSTALL_CJK_FONTS` build-arg gate. **Source is not baked in** — `/app/src` is a host RO mount (line 75 comment, plus `container-runner.ts:340-341`). Plugin partials live as `dockerfilePartials` in `container.json` and are concatenated at *per-agent-group image build time* (`generateAgentGroupDockerfile:569-607`), not at base-image build time.
- **Verdict:** v2's Dockerfile is materially better hygiene (pinned versions, pnpm allowlist, tini, build cache mounts, source-not-baked). PORT the pinned `ARG`s + tini + the "source as RO mount" pattern. PORT the pnpm `only-built-dependencies` allowlist if v1 ever moves off raw `npm install`. Per-agent-group image build is tied to self-mod (SKIP-ARCH).

#### IPC / message piping

- **v1 approach:** File-based IPC under `data/ipc/<group>/`. Host writes follow-up messages to `input/<ts>-<rand>.json` (atomic via `write-then-rename` — `group-queue.ts:194-203`); container polls `IPC_INPUT_DIR` every 500ms (`agent-runner/src/index.ts:331-378`). `_close` sentinel ends the loop. Container also writes to `messages/`, `tasks/` for snapshots.
- **v2 approach:** Two SQLite files per session (`inbound.db` host-write, `outbound.db` container-write). Container polls `messages_in` table every 1000ms idle / 500ms when active (`poll-loop.ts:13-14`). Heartbeat is a file touch at `/workspace/.heartbeat`, intentionally separate from DB to avoid cross-mount lock contention.
- **Verdict:** v2's two-DB IPC is one of the rewrite's central architectural shifts. SKIP-ARCH wholesale.

### What surprised me

1. **`MAX_CONCURRENT_CONTAINERS` is dead code in v2.** It's exported from `/data/nanoclaw-v2/src/config.ts:40` with the same default (5) as v1, but `grep` across all of `src/` and `container/` shows zero readers. Either an oversight during the rewrite or an intentional drop without removing the config. Either way it's a regression: bursty inbound on N sessions = N parallel Chromium-running containers, no host-wide cap. This is the single largest concrete regression I found in this area.

2. **v2 ships zero kernel-level container hardening on Docker.** No `--cap-drop`, no `seccomp`, no `--memory`, no `--pids-limit`, no `--security-opt=no-new-privileges`. v1's hardening block in `container-runtime.ts:156-177` is one of the more security-aware bits of code in either repo, and it's just gone in v2. The v2 container does run as `USER node` (UID 1000, via the Dockerfile) and uses tini for clean PID-1 semantics, but that's not a substitute for cap-dropping a Chromium-driving container. CONTRIBUTE candidate.

3. **The pre-task script hook is more interesting than I expected.** A 30-second bash slot with the container's env + tools allowlist that decides `wakeAgent: bool` lets you build "watcher tasks" cheaply: cron polls every minute, runs `curl … | grep`, only wakes the agent if there's actually news. This pairs naturally with v1's interval-schedule type that v2 dropped — v1 has the schedule shape, v2 has the gate shape. Combining them in nanotars is a natural fit.

### Cross-cutting concerns

- **Concurrency cap (v1 KEEP / v2 CONTRIBUTE)** intersects Area 6 (resource limits). The cap matters more if `--cpus` / `--memory` per container aren't enforced — without limits, an unbounded number of agent containers can saturate the host.
- **Container hardening (PORT to v2 / KEEP on v1)** intersects Area 6 (security model). The cap-drop block, seccomp profile, and resource limits are one logical group; either all stay or none stay. Apple Container's no-op for these is correct (the VM provides the isolation) so the abstraction must persist.
- **Self-modification (SKIP-ARCH)** depends on Area 2 (approvals module + `pending_approvals`) and Area 1 (per-group `container.json`). Cannot be ported in isolation.
- **Provider abstraction (SKIP-ARCH)** depends on Area 1 (`agent_provider` columns on `agent_groups` and `sessions`) and Area 5 (skill-installed branches model for per-provider modules).
- **Pre-task script hook (ADOPT)** depends on a small Area 1 schema change: add `script TEXT NULL` to `scheduled_tasks`. Otherwise self-contained — runs inside the existing per-group container before the agent call.
- **Pause/resume (SKIP-ARCH)** depends on Area 4 (delivery-action registry on outbound DB) and the agent's MCP tool to emit the action. v1's `emergencyStop` keeps working as-is.
- **`task_run_logs` audit table (KEEP)** is referenced by Area 5 (dashboard) — v2 has no equivalent and any UI work that wants per-task history will need it.

### Open questions

1. **`MAX_CONCURRENT_CONTAINERS` upstream — is it deliberately disabled?** Single-line check around `wakeContainer` would re-enable. Worth opening a CONTRIBUTE PR upstream and seeing the response. Confidence on the regression is high; intent is unclear (the constant being preserved suggests it was meant to be wired).
2. **Should the v2 hardening contribution include Apple Container support?** v1's `extraRunArgs()` returns `[]` for Apple Container intentionally. v2 dropped Apple Container entirely. A CONTRIBUTE PR could either (a) re-add the abstraction layer + wire all the flags into Docker only, or (b) just add the Docker flags and leave runtime abstraction as a future. (a) is bigger but more correct; (b) is the immediate-value version.
3. **Pre-task script — should the script run inside the same container as the agent, or in a sibling lightweight container?** v2 runs it in the same container so it inherits secrets/tools. Pro: cheap, simple, no extra spawn. Con: a misbehaving script can wedge the container before the agent ever runs (30s timeout caps that). For nanotars's per-group model, "same container" is the obvious answer; flagging in case the answer is non-obvious for someone else.
4. **Auth-vs-generic split on task failure notify — is `isAuthError` reliable enough?** v1's `router.ts:isAuthError` does pattern matching on error strings. False positives produce the wrong message; false negatives just degrade to the generic path. Worth checking the patterns are still current against the latest SDK error formats before contributing upstream.
5. **The interval-schedule type drop in v2 — accidental or principled?** v2 only handles `cron` in `handleRecurrence`. `interval` and `once` may have been considered redundant with cron (every-N-minutes-cron is `*/N * * * *`), but `once` in particular isn't expressible cleanly as cron. Worth surfacing during the CONTRIBUTE conversation.
