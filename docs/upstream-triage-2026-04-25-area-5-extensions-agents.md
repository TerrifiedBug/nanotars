## Area: Extensions, agent teams, identity, dashboard

### Functional inventory

- **v1-archive (nanotars):** A single, runtime-loaded plugin system is the central extension point. `src/plugin-loader.ts:316-338` discovers plugin directories under `plugins/` (one or two levels deep — `plugins/<name>/` or `plugins/<category>/<name>/`), parses `plugin.json` via `parseManifest()` (`plugin-loader.ts:36-63`), and dynamically imports `index.js` to bind named hook functions (`plugin-loader.ts:355-377`). The `PluginRegistry` (`plugin-loader.ts:150-313`) owns six container-injection mechanisms — env vars, container-skills, container-side SDK hooks, additional read-only mounts, MCP `mcp.json` fragments, and `Dockerfile.partial` partials — plus four host-side hook lifecycles (`onStartup`, `onShutdown`, `onInboundMessage`, `onOutboundMessage`, `onChannel`). MCP fragments are merged into a single per-group file at `data/env/<groupFolder>/merged-mcp.json` and mounted at `/workspace/.mcp.json` (`container-mounts.ts:158-170`). Channels (`channelPlugin: true`) are themselves plugins. Skills are distributed as a Claude Code marketplace (`TerrifiedBug/nanoclaw-skills`) wired via `.claude/settings.json:2-9`. Per-group identity composes from `groups/<folder>/IDENTITY.md` with fallback to `groups/global/IDENTITY.md`, plus `groups/global/CLAUDE.md` for non-main groups (`container/agent-runner/src/index.ts:497-516`). Agent teams live at `groups/<folder>/agents/<name>/{agent.json,IDENTITY.md,CLAUDE.md}` and are discovered inside the container at runtime by `discoverAgents()` (`container/agent-runner/src/index.ts:397-451`), which builds the SDK `agents` map for the Task tool. The dashboard is a fully-fledged ~900 LOC htmx admin UI delivered as a marketplace plugin (`docs/MARKETPLACE.md:42`); webhooks are also plugins (`docs/PLUGINS.md:445-499`).

- **upstream/main (nanoclaw v2):** No plugin loader. Extensions land via two distinct mechanisms. (1) Host-side: a barrel-import + skill-merged-branch model — `src/channels/index.ts:9-10`, `src/providers/index.ts`, `src/modules/index.ts:19-26` each import sibling files for *side-effect* registration calls (`registerChannelAdapter`, `registerProviderContainerConfig`, `registerDeliveryAction`, `registerApprovalHandler`). Each `/add-<name>` skill copies a module file from a long-lived `skill/<name>` git branch into the standard path and appends one import line to the relevant barrel (`docs/skills-as-branches.md:1-200`). (2) Per-group: a `groups/<folder>/container.json` declares `mcpServers`, `packages`, `additionalMounts`, `skills`, `dockerfilePartials`, `envAllowlist` (`src/container-config.ts:33-71`); a per-agent-group Docker image is built by concatenating the listed partial files (`src/container-runner.ts:550-660`). MCP "tools" inside the container live as in-process modules under `container/agent-runner/src/mcp-tools/` and self-register via `registerTools(...)` from a barrel (`mcp-tools/index.ts:1-23`, `server.ts:24-33`). Agent-to-agent messaging is a *delivery* concept — `src/modules/agent-to-agent/` provides a `create_agent` system action (`create-agent.ts:37-126`) and a `channel_type === 'agent'` route (`agent-route.ts:123-167`), persisted as bidirectional rows in `agent_destinations` (`db/agent-destinations.ts:44-109`). Per-group identity is composed at every spawn by `composeGroupClaudeMd()` (`claude-md-compose.ts:42-130`) — emits `CLAUDE.md` as an imports-only file pulling `.claude-shared.md` (symlink to `/app/CLAUDE.md`), per-skill `instructions.md` fragments, MCP-server inline fragments, and the user-editable `CLAUDE.local.md`. Optional `groups/identity/` is mounted at `/workspace/identity` and groups opt in via `@../identity/<file>.md` imports (`container-runner.ts:316-326`). The dashboard is reduced to a one-way `dashboard-pusher.ts` (~590 LOC) that POSTs JSON snapshots and a tail of `logs/nanoclaw.log` to an external `@nanoco/nanoclaw-dashboard` npm package every 60s (`dashboard-pusher.ts:33-46`, install via `/add-dashboard` skill). Webhooks are a built-in shared HTTP server `src/webhook-server.ts:73-124` that routes `/webhook/<adapterName>` to whichever Chat SDK adapter registered itself.

### Implementation comparison

#### Functionality: Extension loading + manifest

- **v1 approach:** Manifest-driven discovery. Every plugin has a typed `plugin.json` (`plugin-types.ts:6-33`) declaring 13 fields including `containerEnvVars`, `publicEnvVars`, `containerHooks`, `containerMounts`, `dependencies`, `channelPlugin`, `authSkill`, `channels`, `groups`, `version`, `minCoreVersion`. `loadPlugins()` scans `plugins/` once at startup and caches a `PluginRegistry` filterable by `(channel, groupFolder)` (`plugin-loader.ts:242-258`). Unknown manifest fields are silently dropped. Plugin removal is `rm -rf plugins/<name>/` (`docs/PLUGINS.md:294-313`). 280 LOC framework cost (manifest parser, registry, hooks engine).
- **v2 approach:** No manifest. Each extension is a TS file that calls a registry function at module top-level; the barrel file is the manifest (`src/channels/index.ts:9-10`, `src/modules/index.ts:19-26`). Plugin scoping (channel/group filter) doesn't exist — every registered module is global. Removal is `git revert` of the merge commit (`docs/skills-as-branches.md:179-194`). The "framework" is one-line per registry: `registerChannelAdapter` (`channels/channel-registry.ts:25`), `registerDeliveryAction` (`delivery.ts:404`), `registerProviderContainerConfig` (`providers/provider-container-registry.ts:45`), `registerApprovalHandler` (`approvals/primitive.ts:59`).
- **Verdict:** Neutral, with a strong context-dependent split. v1's manifest is *more* expressive (per-channel/per-group scoping, public-vs-secret env classification, explicit version compatibility) and supports drop-in install with no source edits. v2's barrel model is *simpler* (no parser, no registry filter cache, no manifest validation) and trivially typesafe — TS catches a missing import at compile time. For 20+ heterogeneous extensions on a single user's fork, v1's scoping pays off; for upstream's 27-skill-but-most-people-install-3 distribution, v2's git-merge model removes the manifest-version-skew problem. **KEEP v1's `PluginRegistry`** for nanotars — replacing it requires rewriting every `plugins/*/plugin.json` consumer in the host. Do *not* port v2's barrel; it costs more than it saves at this scale.

#### Functionality: MCP server config

- **v1 approach:** Per-plugin `mcp.json` fragments are merged at spawn time into one `merged-mcp.json` per group (`plugin-loader.ts:287-312`, `container-mounts.ts:158-170`). Plugin scoping applies — only in-scope plugins contribute fragments to a given group. Root-level `.mcp.json` is the base. Fragments are JSON files committed alongside the plugin.
- **v2 approach:** Two-pronged. (1) External MCP servers (Postgres, Notion, etc.) are declared as inline objects in `groups/<folder>/container.json:mcpServers` (`container-config.ts:17-25`). Each can carry an `instructions` string that gets composed into `CLAUDE.md` as a `.claude-fragments/mcp-<name>.md` (`claude-md-compose.ts:94-101`). (2) Built-in tools (`send_message`, `schedule_task`, `create_agent`, `install_packages`) are *not* MCP servers at all — they are in-process functions under `container/agent-runner/src/mcp-tools/*.ts`, self-registering via `registerTools(...)` (`mcp-tools/core.ts:262`, `agents.ts:66`, `scheduling.ts`, `self-mod.ts`). One stdio MCP server inside the container exposes them all (`server.ts:35-54`).
- **Verdict:** v2 is meaningfully better for built-in tools (zero IPC overhead, types end-to-end, easy to register approval handlers). v1 is meaningfully better for *external* MCP server declarations, because the v2 model — JSON config inside `container.json` per-group — requires editing every group's config when you want a new server, while v1's plugin fragment is set-and-forget across all groups. Hybrid: **PORT v2's mcp-tools in-process pattern** for new built-in tools nanotars adds, but **KEEP v1's plugin `mcp.json` merge** for external servers. Both can coexist — v1's merger writes to `/workspace/.mcp.json`, the in-process tools register themselves at startup.

#### Functionality: Container skills mounting

- **v1 approach:** Each plugin's `container-skills/` is mounted read-only at `/workspace/.claude/skills/<plugin-name>/` (`plugin-loader.ts:77-88`, `container-mounts.ts:107-115`). Per-channel/per-group scoping applies. Skills are committed in the plugin tree.
- **v2 approach:** Two skill homes. Default skills ship in `container/skills/` (mounted RO at `/app/skills`, `container-runner.ts:343-346`); per-group selection is a list in `container.json:skills` (or `'all'`). The runner symlinks the desired subset into `data/v2-sessions/<id>/.claude-shared/skills/` at every spawn (`container-runner.ts:379-435`). Custom per-group skills live at `groups/<folder>/skills/` — visible via the standard Claude Code skill discovery path inside the container.
- **Verdict:** v2's "shared base + per-group selection + per-group overrides" is cleaner and more flexible than v1's "every plugin contributes skills to every container scoped by manifest". For nanotars users with 27 plugins, v2's selection list is genuinely valuable — you can disable noisy skills per group without uninstalling the plugin. **ADOPT v2's three-tier skills pattern** for nanotars: keep plugin-contributed skills (`plugins/<name>/container-skills/`) but add `groups/<folder>/skills/` discovery and the `container.json:skills` selection list. Medium effort.

#### Functionality: Dockerfile partials

- **v1 approach:** Build-time merge. `container/build.sh:25-58` collects every `plugins/*/Dockerfile.partial` (and one nested level), splits the base Dockerfile at `USER node`, and concatenates partials before that line. One image (`nanoclaw-agent:latest`) per host. Adding/removing a plugin requires `./container/build.sh` to rebuild.
- **v2 approach:** Per-agent-group images, built lazily. `container.json:dockerfilePartials` lists project-relative partial paths (`container-config.ts:57-70`). `buildAgentGroupImage()` (`container-runner.ts:610-660`) concats partials with apt/npm package installs, builds a `nanoclaw-agent:<agent-group-id>` image, and writes the resulting tag back to `container.json`. Triggered on demand by the `install_packages` self-mod approval flow (`src/modules/self-mod/`). Path-traversal guarded (`container-runner.ts:617-626`).
- **Verdict:** v2 is genuinely better. Per-group images mean two groups can have different system tools (one with `himalaya`, another with Postgres client) without 200MB of bloat in every spawn. The path-traversal check (`container-runner.ts:617-626`) is also stronger than v1's "anything matching `plugins/*/Dockerfile.partial` is fair game". **ADOPT** the per-agent-group image build path, but defer until self-mod (Area 3) is also adopted — they're tightly coupled. Large effort if migrated cleanly; if you want the small win first, **PORT** just the path-traversal guard back into v1's `build.sh`.

#### Functionality: Agent teams (per-group sub-agents)

- **v1 approach:** `groups/<folder>/agents/<name>/{agent.json,IDENTITY.md,CLAUDE.md}` is a *Claude Code SDK* agents map. `discoverAgents()` (`container/agent-runner/src/index.ts:397-451`) scans inside the container at startup, reads each agent's IDENTITY+CLAUDE, and returns `{description, prompt, model, maxTurns}`. The map is passed to the SDK `query()` call as `agents` (`container/agent-runner/src/index.ts:606`). The lead agent dispatches via the standard Task tool with `subagent_type="research"`. A PreToolUse hook (`index.ts:553-565`) forces `run_in_background: true` for discovered agents so the lead stays responsive. Agents share the *same* container and SDK invocation — no cross-container IPC.
- **v2 approach:** Different concept entirely. `src/modules/agent-to-agent/` is *inter-agent-group* messaging — each "agent" is a separate agent group with its own container. `create_agent` (`create-agent.ts:37-126`) provisions a brand new agent_group row + filesystem and writes bidirectional `agent_destinations` rows so the parent can address it by local name. `channel_type === 'agent'` messages copy content between session DBs (`agent-route.ts:123-167`). Permission is enforced via the destinations table; file attachments are physically copied between session outboxes/inboxes (`agent-route.ts:70-115`).
- **Verdict:** **They solve different problems and v1's agent teams pattern has no v2 equivalent.** v1's discoverAgents is an in-context Task subagent (10s round-trip, shares context, dies with the parent turn). v2's agent-to-agent is an out-of-process peer agent (separate container, persistent, cross-turn). Both are useful. **KEEP v1's `discoverAgents`** unchanged — the v1 file format is exactly what the Claude Agent SDK consumes. Optionally **ADOPT v2's `agent_destinations` + `channel_type='agent'`** as a *separate* feature for long-lived companion agents (see Open Questions below — ADOPT is large because it requires agent_destinations table, write-destinations propagation, and cross-container DB IPC, all of which depend on the v2 entity model).

#### Functionality: Per-group identity composition

- **v1 approach:** Static. Container-side check: `groupIdentityPath = '/workspace/group/IDENTITY.md'` then fall back to `globalIdentityPath = '/workspace/global/IDENTITY.md'` (`container/agent-runner/src/index.ts:497-505`). Concatenate with `globalClaudeMd` for non-main groups (`index.ts:507-512`). The combined string is passed as `systemPrompt.append`. `groups/global/` ships in the repo with a default IDENTITY (`groups/global/IDENTITY.md` — TARS persona, 75/90/95 humor/honesty/discretion). Customization = edit `groups/<folder>/IDENTITY.md` directly.
- **v2 approach:** Host-composed at every spawn. `composeGroupClaudeMd()` (`claude-md-compose.ts:42-130`) writes `groups/<folder>/CLAUDE.md` as an imports-only file: `@./.claude-shared.md` (symlink to `/app/CLAUDE.md`, the global base) plus sorted `@./.claude-fragments/<name>.md` imports for each enabled skill (`*.instructions.md`), each MCP tool module (`*.instructions.md` siblings of source), and each external MCP server with inline `instructions`. Per-group editable content lives in `CLAUDE.local.md` (auto-loaded by Claude Code, written never by host). Optional shared persona files at `groups/identity/` are RO-mounted at `/workspace/identity` (`container-runner.ts:316-326`), groups opt in by adding `@../identity/<file>.md` to `CLAUDE.local.md`.
- **Verdict:** v2's composition is *significantly* more sophisticated. The fragment-import pattern means adding a skill auto-injects its instructions; same for MCP servers; same for built-in modules. The `CLAUDE.local.md` split cleanly separates host-managed (regenerated) and user-managed (preserved) content. v1's two-file fallback works, but every plugin that wants to inject instructions has to either edit the global IDENTITY (lossy), edit the per-group IDENTITY (per-group), or hope the agent reads its skill's SKILL.md (passive). **PORT** the composition pattern in two phases: (a) introduce `CLAUDE.local.md` as the writable file and treat `CLAUDE.md` as host-regenerated; (b) introduce `.claude-fragments/` and let plugins ship `instructions.md` next to their skills. Medium-large effort, high payoff.

#### Functionality: Webhooks

- **v1 approach:** Webhook is a plugin (`docs/PLUGINS.md:445-499`). Plugin's `index.js` exports `onStartup`/`onShutdown`; in `onStartup` it spins up an `http.createServer`, validates a Bearer secret, parses `{source, text}`, and calls `ctx.insertMessage(mainJid, ...)` to inject the message into the host's normal pipeline. The plugin manifest declares `containerEnvVars: ['NANOCLAW_WEBHOOK_URL', 'NANOCLAW_WEBHOOK_SECRET']`. Each webhook plugin is an independent server.
- **v2 approach:** Single shared HTTP server in core. `src/webhook-server.ts:73-124` lazily starts on the first `registerWebhookAdapter(chat, adapterName)` call and routes by URL path: `/webhook/<adapterName>` → `chat.webhooks[adapterName](request)`. Used by Chat SDK channel adapters (Telegram, Slack, Discord webhook variants) — not a generic ingest endpoint. There is no equivalent to v1's "anything can call ctx.insertMessage from an HTTP request".
- **Verdict:** Different scopes. v2's webhook-server is plumbing for Chat SDK adapters (not a user-facing capability). v1's webhook plugin is a user-facing "external system → admin chat" ingest. **KEEP v1's webhook plugin** (it's just a plugin); **PORT v2's `webhook-server.ts`** if/when nanotars adopts Chat SDK channel adapters (Area 4). Otherwise the two are non-overlapping.

#### Functionality: Dashboard

- **v1 approach:** A full-featured admin web UI shipped as `nanoclaw-dashboard` marketplace plugin (~900 LOC htmx app per docs/MARKETPLACE.md:42). Per the marketplace listing it surfaces agent groups, sessions, messages, scheduled tasks, channel status, queue state, and registered plugins. It is a host-side plugin running its own HTTP server in the same Node process. Direct DB read access via `PluginContext` (chats, tasks, queue, plugins).
- **v2 approach:** Split into two pieces. (1) Pusher-only `src/dashboard-pusher.ts` (~590 LOC) collects a wide JSON snapshot every 60s — agent groups (with destinations, members, admins, wirings), sessions, channels (live + registered + per-mg), users (with privilege classification), per-model+per-group token usage parsed from JSONL transcripts, context-window usage per agent, hourly inbound/outbound activity buckets for the last 24h, and the most recent 50 messages per session — and POSTs them to `127.0.0.1:<port>/api/ingest` along with a 2s tail of `logs/nanoclaw.log` to `/api/logs/push` (`dashboard-pusher.ts:122-141, 80-119`). (2) The actual UI is an external npm package `@nanoco/nanoclaw-dashboard` installed by `/add-dashboard` skill — separated repo, separated lifecycle.
- **Verdict:** Architecturally cleaner in v2 (the UI is decoupled from the host process, so the dashboard's bugs/restarts don't take down message routing). But the v2 *pusher* loses none of the surfaces v1 had — token usage parsing from JSONL, context-window math, per-channel activity buckets, etc. — and adds a few v1 didn't track (per-model breakdown, context utilization %). **KEEP v1's dashboard plugin** (it works, it's used, it's already the heaviest item in the fork). **CONTRIBUTE** the dashboard to upstream as a v2-compatible adaptor (the v2 pusher is already most of the way there — what v1 contributes is the htmx UI and any analytics v2's pusher doesn't track). High value to upstream, medium effort to adapt.

### What surprised me

1. **v2's "agent-to-agent" module is *not* an upgrade of v1's agent teams — they solve unrelated problems.** From the spec wording I expected a refinement; it's a substitution. v1's `discoverAgents` is the Claude Agent SDK's native `agents` map (in-context, single-container, Task-tool-based); v2's agent-to-agent is cross-container peer-to-peer messaging via the destinations table. Both are valuable. The triage note "v2's agent-to-agent module is a meaningful equivalent" should be flagged false — they're orthogonal and a careful fork would keep both.

2. **v2's `composeGroupClaudeMd` is the most interesting architectural improvement in this area.** I expected the identity comparison to be a wash (both have a global file + per-group override). It isn't. v2's host-regenerated `CLAUDE.md` with `.claude-fragments/` symlinks means every skill/MCP-tool/MCP-server can ship its own `instructions.md` and the agent gets it injected automatically — no manual SKILL.md reading, no plugin index.js to register a snippet. v1's plugin system has six container-injection mechanisms but no equivalent for "always-in-context agent-side instructions". This is a strict win for v2 and the cleanest port candidate in the area.

3. **The dashboard split (heavy plugin → light pusher + external package) is more architecturally sound than I expected.** I went in assuming v2's dashboard reduction was a feature regression. Reading `dashboard-pusher.ts` shows it actually *adds* surfaces v1's dashboard didn't have (per-model token breakdown, context-window utilization %, hourly activity buckets), and decoupling the UI from the host process is a real reliability gain — if the dashboard crashes, message routing keeps working. The v1 dashboard plugin is still better as an *out-of-the-box experience* (no separate npm package install, no port wiring), but the v2 pusher's data model is the better foundation. A CONTRIBUTE PR that wires v1's UI to consume v2-style snapshots could be the best of both.

### Cross-cutting concerns

- **Area 3 (Runtime) — per-agent-group images.** `buildAgentGroupImage` lives in `container-runner.ts` but only fires from the self-mod approval flow (`src/modules/self-mod/apply.ts`). Adopting per-group images in nanotars therefore requires Area 3's self-mod port to land first, or a manual trigger.

- **Area 4 (Channels) — Chat SDK webhook server.** `src/webhook-server.ts` only runs if Chat SDK channel adapters register webhook routes. Whether it's needed depends entirely on Area 4's verdict on adopting Chat SDK adapters.

- **Area 6 (per-group config) — `container.json` schema.** Multiple verdicts in this area (per-group skill selection, dockerfilePartials, mcpServers with `instructions`) all hinge on adopting `container.json` as a per-group config file. If Area 6 declines that schema, the skills-selection and partials-list pieces of this area become dependent on a different storage decision (e.g. per-group `.env` flag).

- **Area 2 (permissions) — `agent_destinations` ACL.** v2's create_agent flow uses `agent_destinations` rows as both routing map and permission grant. If Area 2 ports the user_roles model but not agent_destinations, the agent-to-agent verdict's SKIP-ARCH stays correct.

- **Area 1 (DB) — `agent_destinations` central table.** Same dependency, different layer. Adopting v2's agent-to-agent requires the central-DB table to exist.

- **Area 6 (group-env) — `envAllowlist`.** v1's `containerEnvVars` per-plugin manifest is the analog of v2's `envAllowlist` per-group `container.json` entry. If Area 6 ports `group-env`, decide whether plugin-declared `containerEnvVars` still feed in or are subsumed.

### Open questions

1. **For per-group images, is the `nanoclaw-agent:<agent-group-id>` tag scheme correct under nanotars's slug-based image-naming policy?** v2's `setup/lib/install-slug.sh` derives an image name from project root, allowing two installs on the same host. Nanotars doesn't have this policy today. If the image-tag base differs between installs, per-group images may need a slugged base. Needs human decision on whether to port the slug logic alongside.

2. **Should the agent-team CONTRIBUTE PR upstream introduce v1's `groups/<folder>/agents/<name>/` literally, or wrap it in v2's per-group `container.json`?** Upstream's preference for `container.json`-as-single-source-of-truth (`container-config.ts`) might conflict with v1's filesystem-per-agent layout. A PR could either add `agents` as a dict in `container.json` (more v2-idiomatic) or ship `discoverAgents` as a v2 container-runner addition (more direct port).
