# Phase 4.5: Supply-chain hardening (pnpm migration) — Design Spec

**Goal:** Migrate the v1-archive root package from npm to pnpm with `minimumReleaseAge: 4320` (3-day registry hold) + `onlyBuiltDependencies` allowlist, mirroring v2's supply-chain policy. Retain npm for the in-container `agent-runner/` subpackage. Add a "Supply Chain Security" section to `CLAUDE.md`.

**Why now:** Phase 4.5 sits between RBAC (Phase 4) and self-modification (Phase 5). Phase 5 introduces agents installing packages dynamically into per-group images — that's where `minimumReleaseAge` actually pays off. Landing pnpm here means Phase 5 is born under the policy from day one. Phase 4 added zero runtime deps so no lockfile churn yet.

**Source upstream:** `qwibitai/nanoclaw` v2 (`/data/nanoclaw-v2/`). Triage doc: `/data/nanotars/docs/upstream-triage-2026-04-25.md` (Phase 4.5 / Cluster G section). Triage decision: ADOPT v2 → v1, medium effort, medium impact.

## Key decisions

### `onlyBuiltDependencies = [better-sqlite3]` only

v2 lists `[better-sqlite3, esbuild, protobufjs, sharp]`. v1-archive root deps are: `@onecli-sh/sdk`, `better-sqlite3`, `cron-parser`, `pino`, `pino-pretty`, plus dev deps. Of these, only `better-sqlite3` has a native binding compile step. `esbuild`/`protobufjs`/`sharp` aren't direct or transitive deps in v1-archive root. We add only what we use — narrowest allowlist consistent with v2's policy.

If a future dep needs a build script, the human must approve adding it (per v2's CLAUDE.md rule).

### `pnpm import` not `pnpm install --lockfile-only`

`pnpm import` reads `package-lock.json` and produces `pnpm-lock.yaml` with the **same resolved versions**. This is critical: a fresh `pnpm install` would re-resolve from `package.json` ranges and could drift. The previous Phase 3 implementer ran a fresh `pnpm install` accidentally and clobbered native bindings (107 test failures). This time we explicitly preserve versions.

### Global installs in Dockerfile move to pnpm

Currently:
```dockerfile
RUN npm install -g agent-browser@${AGENT_BROWSER_VERSION} @anthropic-ai/claude-code@${CLAUDE_CODE_VERSION}
RUN npm install -g tsx@${TSX_VERSION}
```

Becomes:
```dockerfile
RUN npm install -g pnpm@9.15.0
RUN pnpm install -g agent-browser@${AGENT_BROWSER_VERSION} @anthropic-ai/claude-code@${CLAUDE_CODE_VERSION}
RUN pnpm install -g tsx@${TSX_VERSION}
```

(npm bootstraps pnpm — pnpm itself is published by a long-established maintainer and a 3-day hold on it would create a chicken-and-egg.) The agent-runner `npm install` line stays untouched. Agent-runner is a separate package tree that does not need pnpm policy applied — it's installed inside the container at image build time, not by the host.

`.npmrc` with `minimumReleaseAge=4320` already provides the supply-chain hold for npm-invoked installs (including the agent-runner `npm install` and any `npm install -g pnpm@…` bootstrap), so dropping pnpm into the Dockerfile doesn't lose policy coverage on the npm side.

### `pnpm/action-setup` pinned version

Pin `pnpm/action-setup@v4` (current major) with `version: 9.15.0` in the `with:` block. Specific version chosen to match what's published as of 2026-04-26 and within what npm's `minimumReleaseAge=3d` would allow. The implementer confirms the latest stable pnpm 9.x at execution time and pins to that. CI cache key is `pnpm` (not npm).

### CI: split jobs stay split

`host` + `bash-tests` jobs run pnpm. `agent-runner` job runs npm (it's installing inside `container/agent-runner/` which has its own `package-lock.json`). This matches the runtime-split convention of v2 (host = pnpm, container = different runtime).

### `package.json` script audit

Current scripts use `npm run typecheck && npm run test && npm run test:bash` in `test:all`. Under pnpm, these still work because pnpm intercepts `npm run X` calls within scripts (the running pnpm passes its bin dir to the script's PATH, but the literal command is still `npm run …`). However, idiomatic pnpm uses `pnpm run X`. We can leave scripts alone — they work either way, and leaving them avoids gratuitous churn. Manager-agnostic chains are fine.

If something breaks, the fallback is to change `npm run` → `pnpm run` in `test:all`.

## Risks + mitigations

| Risk | Mitigation |
|------|------------|
| `pnpm install` re-resolves and clobbers native bindings (Phase 3 lesson) | Use `pnpm import` to preserve versions exactly, then `pnpm install --frozen-lockfile` ever after. Sanity test runs all tests after migration. |
| `pnpm install -g` in Dockerfile creates a global root pnpm has to know about | pnpm's global root is per-user and persists; switching from npm-global → pnpm-global works in a fresh image build. We test by rebuilding the container. |
| CI's `cache: 'pnpm'` requires pnpm to be installed before `setup-node` | `pnpm/action-setup` runs before `setup-node` per pnpm docs. Order is: checkout → action-setup → setup-node (with pnpm cache) → install. |
| Dockerfile's `RUN npm prune --omit=dev` on line 71 — does that still work? | That's inside `container/agent-runner/` workdir scope (after `WORKDIR /app` + `COPY container/agent-runner/package*.json ./` + `RUN npm install`). It's an npm command on an npm-installed tree. Stays unchanged. |
| Agent-runner's `npm install` inside the container relies on the `.npmrc` `minimumReleaseAge=4320` | Confirmed — `.npmrc` is the fallback for npm-direct invocations, exactly the case here. No change needed. |
| Memory file says "pnpm install clobbers native bindings" — operators reading it may be confused after migration | Update the memory file at the end of the phase. |

## Rollback plan

If `pnpm install` produces a non-working tree mid-phase, the rollback is a single revert per task:

1. `git revert <commit-sha>` for the offending task.
2. If the revert chain crosses Dockerfile + lockfile + workspace, revert all the way back to the start-of-phase HEAD: `git reset --hard <pre-phase-sha>` then re-apply only the parts that worked.
3. The `.npmrc` and `package.json` are safe — the `.npmrc` predates this phase (Phase 1) and `package.json` is only nudged on scripts.

Each task ships independently and leaves a working build, so rollback is bounded.

## Validation gates

At the end of the phase:

1. `pnpm install --frozen-lockfile` clean (no errors, no missing native binding rebuild).
2. `pnpm test` matches baseline test count (839 host).
3. `pnpm run build` succeeds.
4. `pnpm run typecheck` clean.
5. `./container/build.sh` succeeds (image rebuilds, agent-runner npm install + pnpm globals all work).
6. `git status` clean — no stray `package-lock.json` reappearing, no untracked node_modules.
7. CI green on push (workflow runs pnpm/action-setup + frozen-lockfile install).

## Open questions (decided autonomously)

- **`onlyBuiltDependencies` allowlist size:** decided `[better-sqlite3]` only — narrowest viable allowlist. v2's larger list reflects v2's larger dep graph.
- **pnpm major version:** decided pnpm 9.x (current stable). Implementer pins exact patch at execution time.
- **CI `cache: 'pnpm'` key:** decided yes. Speeds up CI without changing semantics.
- **Should agent-runner ALSO migrate to pnpm?** decided no. Stays on npm. Agent-runner is a separate package installed inside the container by the existing `npm install` step. Adding pnpm there would mean either (a) bootstrapping pnpm in the container image just for one install, or (b) making agent-runner a pnpm workspace member, which conflicts with its role as a self-contained tree shipped via Dockerfile. The `.npmrc` `minimumReleaseAge=4320` already gives it the 3-day hold. No upside.
- **Global Dockerfile installs (claude-code, agent-browser, tsx):** decided migrate to `pnpm install -g`. Otherwise pnpm policy doesn't cover these critical dev binaries. The chicken-and-egg is solved by `npm install -g pnpm` first (which itself is gated by `.npmrc minimumReleaseAge`).
