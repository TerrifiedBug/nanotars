## Area: Persistence layer

### Functional inventory

- **v1-archive (nanotars):** A single `better-sqlite3` central DB file at `<STORE_DIR>/messages.db` holds **everything**: per-chat metadata (`chats`), full message history (`messages`), `scheduled_tasks` + `task_run_logs`, plus the host-only key/value tables `router_state`, `sessions`, and `registered_groups`. Schema is created in one block in `src/db/init.ts:21-90`, with a hand-rolled migration system (`schema_version` + a hardcoded `MIGRATIONS` array of 5 ALTER-TABLE migrations at `src/db/init.ts:103-127`, including a sentinel-detect path for older databases that pre-date the migration table at `src/db/init.ts:140-160`). Domain accessors live under `src/db/{messages,state,tasks}.ts`; everything goes through a module-level `getDb()` singleton (`src/db/init.ts:10-14`). Container-bound state is shipped via two host-written JSON snapshots (`current_tasks.json`, `available_groups.json`) under `data/ipc/<group>/` — DB rows mapped to JSON in `src/snapshots.ts:8-19,21-48,62-88`. There is also a one-time JSON-to-SQLite migration of pre-DB state (`router_state.json`, `sessions.json`, `registered_groups.json`) at `src/db/migrate.ts:12-63`, plus an online-backup hook keeping the two latest backups (`src/db/init.ts:214-240`). Reactions and message lookups all go through `messages` keyed on the `(id, chat_jid)` composite (`src/db/init.ts:36-37`).

- **upstream/main (nanoclaw v2):** Three-DB model: one **central DB** (`data/v2.db`) for entity model + wiring + cross-session tables, plus **per-session pairs** (`inbound.db` + `outbound.db`) under `data/v2-sessions/<session_id>/`. Central schema is canonicalized in `src/db/schema.ts:7-147` and constructed via a versioned migration framework (`src/db/migrations/index.ts:36-73`, currently 13 numbered + 3 module-owned migrations) keyed on `schema_version.name` rather than `version` so install skills can add module migrations without coordinating numbers. Central tables: `agent_groups`, `messaging_groups`, `messaging_group_agents`, `users`, `user_roles`, `agent_group_members`, `user_dms`, `sessions`, `pending_questions`, `pending_sender_approvals`, `unregistered_senders`, `chat_sdk_*` (KV / locks / lists / subscriptions for the Chat SDK adapter, `src/state-sqlite.ts:32-181`). Per-session tables (`src/db/schema.ts:158-258`): `messages_in`, `delivered`, `destinations`, `session_routing` on host-owned `inbound.db`; `messages_out`, `processing_ack`, `session_state`, `container_state` on container-owned `outbound.db`. Each file has **exactly one writer** to eliminate cross-mount lock contention; visibility relies on `journal_mode = DELETE` on inbound.db (WAL's `-shm` is mmap-based and VirtioFS does not propagate mmap coherency — see `container/agent-runner/src/db/connection.ts:12-19`). Seq numbers are partitioned: host writes even, container writes odd, picked by `nextEvenSeq` / `writeMessageOut` (`src/db/session-db.ts:81-84`, `container/agent-runner/src/db/messages-out.ts:45-77`). State helpers: `session_state` keyed per-provider (`container/agent-runner/src/db/session-state.ts:14-79`) lets the agent's continuation survive restarts and provider flips losslessly; `container_state` (single-row id=1) tracks the in-flight tool so the host sweep can widen stuck-detection while Bash runs with a long declared timeout (`container/agent-runner/src/db/connection.ts:86-115`).

### Implementation comparison

#### Functionality: central-DB initialization & connection
- **v1 approach:** `initDatabase()` opens `messages.db`, sets `journal_mode=WAL`, calls `createSchema()` which inlines DDL + runs migrations, then runs JSON-import + log-prune startup tasks and triggers an online backup (`src/db/init.ts:177-190`). A module-scoped `let db` and `getDb()` accessor (`src/db/init.ts:10-14`). Test variants `_initTestDatabase()` create an in-memory DB and re-run `createSchema` (`src/db/init.ts:193-202`).
- **v2 approach:** `initDb(dbPath)` only opens the file, sets `WAL` and `foreign_keys=ON`, and returns the handle — schema creation is a separate concern handled by the migration runner. Singleton lives in `src/db/connection.ts:7-21`. Adds `hasTable(db, name)` (`src/db/connection.ts:43-48`) used by core tables that need to gracefully degrade when an optional module's table isn't installed yet.
- **Verdict:** v2 is cleaner — separating connection from schema (and using a real migration list) is the right shape. v1's `createSchema` doing both the inlined DDL *and* the migration loop in one function makes it easy to drift the inline DDL away from migrations. v1's `_initTestDatabaseFrom()` (`src/db/init.ts:199-202`) still re-runs `createSchema` against an existing DB which is slightly weird (it's idempotent only because of `IF NOT EXISTS`).

#### Functionality: schema migrations
- **v1 approach:** Hand-rolled `MIGRATIONS` array (`src/db/init.ts:103-127`) with `safeAddColumn` swallowing duplicate-column errors (`src/db/init.ts:96-100`); inserts a sentinel record for pre-system DBs by pragma-inspecting `messages`, `scheduled_tasks`, `registered_groups` (`src/db/init.ts:140-160`). `schema_version (version TEXT PRIMARY KEY, applied_at TEXT)`. All migrations are pure ALTER TABLE — no compound migrations, no module installation hook.
- **v2 approach:** Numbered migration files under `src/db/migrations/<NNN>-<name>.ts` exporting `{ version, name, up }` (`src/db/migrations/index.ts:16-20`). Runner uses `schema_version.name` as the de-dup key with auto-assigned applied-order `version` (`src/db/migrations/index.ts:36-73`) — explicitly to allow module migrations to pick arbitrary numbers without coordinating. Each migration runs in its own transaction. Module migrations have a different naming convention (`module-<area>-<table>`).
- **Verdict:** v2's framework is materially better for a multi-skill, multi-branch install model. v1's array-with-sentinels works for a single-author repo but doesn't survive the channel/provider skill model. **PORT** the framework if v1 ever picks up the install-skill model; for now KEEP if v1's plugin approach is staying file-based. See cross-cutting note for Area 2.

#### Functionality: message storage
- **v1 approach:** Cross-channel `messages` table keyed on `(id, chat_jid)` composite (`src/db/init.ts:36-37`); rich content fields including `content`, `reply_context` (JSON), `is_bot_message`, `is_from_me`. `storeMessage()` upserts with `INSERT OR REPLACE` and emits a `dbEvents.emit('new-message', chatJid)` (`src/db/messages.ts:95-110`). `getNewMessages()` filters out bot messages with both the boolean column and a content-prefix backstop using `${botPrefix}:%` (`src/db/messages.ts:146-158`). Every registered chat's message history lives forever in this single table.
- **v2 approach:** No central message-history table at all. Inbound messages live only in the per-session `messages_in` table (`src/db/schema.ts:159-176`) with status tracking (`pending` / `processing` / `completed` / `failed`), and once status flips to `completed` the row sticks around but is unused outside of debugging. There is no global `messages` aggregation — message history *as recall context* is the agent's responsibility (`/workspace/agent/conversations/`), not the DB's.
- **Verdict:** Architecturally different. v1's design assumes a single shared message store the host can re-read for batching/recap (`getRecentMessages`); v2 punts recall to the agent's filesystem. v1's bot-message backstop using a content prefix is a smell — fragile across versions of the assistant name. **SKIP-ARCH:** v2's design depends on session DBs and the per-session container model; can't port without the full v2 split.

#### Functionality: per-chat metadata / chat list
- **v1 approach:** `chats(jid PK, name, last_message_time)` populated via `storeChatMetadata()` (`src/db/messages.ts:8-33`). Includes magic `__group_sync__` row used as a "last sync timestamp" (`src/db/messages.ts:73-89`).
- **v2 approach:** `messaging_groups` (`src/db/schema.ts:26-36`) is the conceptual equivalent — but it stores routing metadata (channel_type, platform_id, name, is_group, unknown_sender_policy), not last-activity. Last-activity tracking has been replaced by per-session `last_active` on `sessions` (`src/db/schema.ts:113`).
- **Verdict:** Different responsibilities; v2's approach is cleaner — `messaging_groups` is identity, `sessions` is liveness. The `__group_sync__` magic-row pattern is a v1 nanotars hack that should be deleted in any port. **Neutral / SKIP-ARCH** for the table shape; the magic-row hack is a CONTRIBUTE-as-cleanup if v1 keeps the table.

#### Functionality: scheduled tasks
- **v1 approach:** Full table `scheduled_tasks` (`src/db/init.ts:42-58`) with first-class fields: `schedule_type`, `schedule_value`, `next_run`, `last_run`, `last_result`, `status`, `model`, `context_mode` ('group' | 'isolated'), and FK-joined `task_run_logs` (`src/db/init.ts:60-69`). Comprehensive accessor surface: `claimTask` clears `next_run` to prevent re-enqueue (`src/db/tasks.ts:102-104`), `getDueTasks`, `updateTaskAfterRun` auto-flips status to 'completed' when `nextRun IS NULL` (`src/db/tasks.ts:119-132`), `pruneTaskRunLogs` keeps the table bounded (`src/db/tasks.ts:171-176`), and `getRecentTaskRunLogs` JOINs with the task table for the dashboard (`src/db/tasks.ts:141-152`).
- **v2 approach:** No scheduled_tasks table. Scheduling lives entirely inside `messages_in` via `process_after`, `recurrence`, `series_id`, and `trigger=0` (accumulated context) columns (`src/db/schema.ts:159-176`). A scheduled task is just a `messages_in` row with `process_after` set; recurrences are re-inserted by host-sweep code in `src/modules/scheduling/` (out of scope for this area, see Area 3). No equivalent of `task_run_logs` — historical run records aren't retained as a first-class entity.
- **Verdict:** v1 wins on observability — `task_run_logs` + JOIN view is a clean dashboard story. v2 wins on architectural tidiness — scheduling is "just another inbound message," no separate state machine. **CONTRIBUTE** v1's `task_run_logs` structure if upstream cares about scheduled-task observability; **PORT** v2's `process_after` + `series_id` + `trigger=0` accumulator pattern *if* v1 ever needs accumulated context with delayed wake. The `context_mode` ('group' vs 'isolated') is a v1-unique concept that doesn't map onto v2 at all (v2's session_mode covers a different axis).

#### Functionality: registered groups / messaging-group routing
- **v1 approach:** `registered_groups (jid PK, name, folder UNIQUE, trigger_pattern, added_at, container_config JSON, requires_trigger, channel)` (`src/db/init.ts:80-89`). `getAllRegisteredGroups()` returns a record keyed by JID (`src/db/state.ts:115-128`). `isValidGroupFolder` defends against path-traversal from corrupted DB rows (`src/db/state.ts:74-79`) — explicit allowlist regex + reserved-folder set (`'global'`).
- **v2 approach:** Split across three tables: `agent_groups` (workspace identity), `messaging_groups` (chat identity), `messaging_group_agents` (the M:N wiring) — `src/db/schema.ts:12-56`. `trigger_pattern + requires_trigger` were superseded by four orthogonal columns: `engage_mode` ('pattern'/'mention'/'mention-sticky'), `engage_pattern`, `sender_scope` ('all'/'known'), `ignored_message_policy` ('drop'/'accumulate') (`src/db/schema.ts:42-56`, migration `src/db/migrations/010-engage-modes.ts`). Container config is no longer a JSON column — it lives on disk at `groups/<folder>/container.json`. No folder-validation regex on the DB read path (folder is constrained at `agent_groups.folder UNIQUE` and validated at create time elsewhere).
- **Verdict:** v2's separation of identity vs. wiring is the structural improvement of the rewrite. v1's `requiresTrigger + trigger_pattern` is exactly the conflated axis that migration 010 explicitly factored apart. **SKIP-ARCH** wholesale (entity model is the rewrite), but **PORT-the-design-lesson:** the four-axis engage model is a much clearer schema. The `isValidGroupFolder` defensive validator (`src/db/state.ts:74-79`) is something v2 *lacks* in its central DB read path — minor **CONTRIBUTE** candidate (defense-in-depth against corrupted rows).

#### Functionality: sessions / per-session container state
- **v1 approach:** `sessions(group_folder PK, session_id)` (`src/db/init.ts:76-79`) — a single string per group, just enough to resume the Claude SDK session. `getAllSessions()` returns a flat record of folder→sessionId (`src/db/state.ts:35-44`). No per-session state table; runtime state (idle, queued) is in-memory only.
- **v2 approach:** Full sessions table with status, container_status, last_active, agent_provider, plus a M:N relation to messaging_group + thread (`src/db/schema.ts:105-117`). Each session also has its own pair of DB files (`inbound.db` + `outbound.db`) — see Area 3 for the lifecycle. Per-provider continuation IDs live in `outbound.db.session_state` keyed `continuation:<provider>` (`container/agent-runner/src/db/session-state.ts:14-79`) so flipping providers is lossless. There's even a one-time `migrateLegacyContinuation` (`container/agent-runner/src/db/session-state.ts:52-67`) that adopts the pre-per-provider single-key value into the current provider's slot.
- **Verdict:** v2 is materially richer; v1's sessions table holds the bare minimum (just the SDK resume token). **SKIP-ARCH** as a wholesale port (depends on the per-session container model in Area 3); but the lesson `session_state` keyed-per-provider with explicit continuation namespacing is a **PORT-as-design** if v1 ever adds non-Anthropic providers.

#### Functionality: router state / generic KV
- **v1 approach:** `router_state(key TEXT PK, value TEXT)` accessed via `getRouterState`/`setRouterState` (`src/db/state.ts:7-18`). Used to hold things like `last_timestamp` (poll cursor), `last_agent_timestamp` (per-agent JSON-encoded). Two values total in practice.
- **v2 approach:** No general-purpose central KV. Instead two specialized tables: `chat_sdk_kv` (with TTL, `src/state-sqlite.ts:32-69`) and per-session `session_state` (no TTL, `container/agent-runner/src/db/session-state.ts:20-35`). The `chat_sdk_*` family is a full StateAdapter implementation: KV with TTL, locks (`acquireLock`/`extendLock`/`forceReleaseLock`, `src/state-sqlite.ts:87-117`), lists (with maxLength + TTL trimming, `src/state-sqlite.ts:121-136`), and queues (built on lists, `src/state-sqlite.ts:150-172`). Periodic `cleanup()` purges expired rows on every `connect()` and on every `get` (`src/state-sqlite.ts:176-181`).
- **Verdict:** v2's `SqliteStateAdapter` is a much more complete persistence primitive — TTL, locks, queues are the right toolset for distributed-coordination work. v1's `router_state` is a text-blob pair sufficient for two values. The Chat SDK lock implementation (compare-and-swap via token UUID, `src/state-sqlite.ts:87-117`) is genuinely useful and not specific to chat-sdk. **PORT** the lock pattern + TTL'd KV as a generic primitive into v1 if v1 ever needs distributed coordination. Note: cleanup-on-every-`get` is paranoid — empirically fine but a periodic background sweep would be cheaper at scale.

#### Functionality: message-status tracking (host ↔ container)
- **v1 approach:** N/A in DB. Container is invoked synchronously via stdin/stdout (see Area 6), and message-status tracking is purely in-process. The message store is read-only from the container's perspective — there's no acknowledgment ever written back.
- **v2 approach:** Two-side state machine via `processing_ack` (container-owned, `src/db/schema.ts:233-237`). Container writes `processing` on pickup (`container/agent-runner/src/db/messages-in.ts:81-90`), `completed`/`failed` on finish; host reads it via `syncProcessingAcks` to flip the inbound row's status (`src/db/session-db.ts:148-161`) and via `getStuckProcessingIds`/`getProcessingClaims` to detect crashes (`src/db/session-db.ts:163-181`). On container startup `clearStaleProcessingAcks` discards any `processing` rows from the previous run so the new container re-processes (`container/agent-runner/src/db/connection.ts:141-143`). `delivered` table on inbound.db is the dual: host-written record of which outbound IDs have been delivered, with optional platform_message_id mapping for reactions/edits (`src/db/schema.ts:180-185`, `src/db/session-db.ts:237-256`).
- **Verdict:** v2's pattern is the load-bearing IPC layer (replaces v1's stdin/stdout). **SKIP-ARCH** — the whole pattern is meaningless without the per-session container model. Worth understanding as background for Area 3.

#### Functionality: snapshots / IPC of state to container
- **v1 approach:** Host writes filtered JSON snapshots into `data/ipc/<group>/`: `current_tasks.json` and `available_groups.json` (`src/snapshots.ts:21-48,62-88`), atomic via tmp+rename. Main-group sees everything; non-main groups see only their own filtered slice. Container polls these files. This is part of the broader file-IPC pattern (see Area 6).
- **v2 approach:** No JSON snapshots. The container reads the live `inbound.db.destinations` table (host overwrites on every wake AND on demand, `src/db/session-db.ts:58-68`); routing defaults come from `inbound.db.session_routing` single-row table (`src/db/session-db.ts:35-47`). Mid-session destination updates take effect without a container restart because the container queries the DB live on every tool call.
- **Verdict:** v2 is simpler and the live-update behavior is genuinely better (v1's snapshots are stale until next poll, and require explicit re-write on every change). **SKIP-ARCH** for direct port; the design lesson — replace JSON-snapshot IPC with "host writes a row, container reads it" — is the entire point of v2's two-DB model.

#### Functionality: backups
- **v1 approach:** `backupDatabase()` calls `db.backup(backupPath)` (better-sqlite3's online backup API) on every `initDatabase()`, keeping the two most recent (`src/db/init.ts:214-240`). Async, non-blocking.
- **v2 approach:** No backup hook. v2 relies on the user to back up `data/v2.db` and `data/v2-sessions/` externally.
- **Verdict:** v1 has a feature v2 lacks. The two-most-recent retention is conservative but useful for recovery. **CONTRIBUTE** to v2 — small, mechanical, no architectural dependency. (Note: v2 has more DB files now, so the backup loop would need to walk `data/v2-sessions/` too; complicates the port slightly.)

#### Functionality: reaction lookups / message-by-id
- **v1 approach:** `getMessageMeta(messageId, chatJid)` returns `{ sender, isFromMe }` for reaction routing (`src/db/messages.ts:204-213`). Lookup keyed on the composite PK directly.
- **v2 approach:** No equivalent. Reactions are routed via `getMessageIdBySeq` / `getRoutingBySeq` (`container/agent-runner/src/db/messages-out.ts:93-144`) which look up by *seq* (the agent-facing virtual ID), and for inbound chat-sdk messages the original 2-part composite is preserved verbatim in the JSON `content.id` field — read out at `messages-out.ts:99-108`. There's a recent fix (commit `5e93609 fix(reactions): use canonical chat-sdk composite id for inbound lookups`) that addresses exactly this issue.
- **Verdict:** Completely different message-ID models. v1's composite PK is simpler; v2's seq-then-content-id chain is necessary because seq is what tools see. Neither side is generally better — they fit their respective architectures. **SKIP-ARCH**.

#### Functionality: dropped-message / unregistered-sender tracking
- **v1 approach:** N/A — v1 does not track senders that were dropped before reaching an agent.
- **v2 approach:** `unregistered_senders` table (`src/db/migrations/008-dropped-messages.ts:9-22`) keyed on `(channel_type, platform_id)` with message_count, first_seen, last_seen, plus an upsert-with-coalesce that increments count on each repeat (`src/db/dropped-messages.ts:16-38`). Used by the router for diagnostics + admin "who has been DM'ing me?" view.
- **Verdict:** **ADOPT** — small, useful, no architectural dependency. The table is keyed on the channel/platform tuple which v1 already has in `chats(jid)`. Worth ~30 LOC port.

### What surprised me

1. **v2 dropped the global message store entirely.** v1's `messages` table — the canonical place to find "what did the user say at 3pm last Tuesday" — has no analog in v2 central DB. Once a `messages_in` row's status is `completed`, it sits idle on disk inside the per-session DB, and there's no aggregate view across sessions. Recall context is fully delegated to the agent's filesystem (`/workspace/agent/conversations/`, see `container/CLAUDE.md`). This is a much bigger philosophical break than I expected — v1 treats history as host-owned data, v2 treats it as agent-owned files.

2. **v2 has two completely separate persistence frameworks running side by side.** The "real" central DB uses better-sqlite3 named-parameter inserts (`@field`) and the migration framework. The Chat SDK adapter (`src/state-sqlite.ts`) uses positional parameters and writes to its own `chat_sdk_*` tables that are *also* in the same `data/v2.db` file. They don't share migration ownership cleanly — `chat_sdk_*` tables come from migration 002 but the adapter's behavior is effectively a self-contained subsystem. Reads "ported from feat/chat-sdk-integration branch" header (`src/state-sqlite.ts:5`) and that's exactly how it feels: bolted on.

3. **Migration 010 is essentially v2 admitting v1's `requires_trigger + trigger_pattern` was the right idea, but implemented wrong.** v2's initial schema (migration 001) had `trigger_rules TEXT` (opaque JSON) + `response_scope TEXT DEFAULT 'all'` — a worse model than v1's two simple columns. Migration 010 then explodes it into four orthogonal columns with explicit semantics for both scope axes. So v2 has actually iterated *toward* the v1 design space, just with a richer 4-axis model than v1's 2-axis one. This means a v1→v2 port of the engage model is plausible without a full architectural cutover (graded SKIP-ARCH→PORT in the matrix).

### Cross-cutting concerns

- **Migrations framework + permissions tables (Area 2):** Migration 001 creates `users`/`user_roles`/`agent_group_members`/`user_dms` (`src/db/migrations/001-initial.ts:41-83`) — all permission-system tables. Their CRUD lives outside this area in `src/modules/permissions/db/*.ts`. Any "port the migration framework" decision has to land before any individual permission-table port. Flagged for Area 2: `pending_sender_approvals` (DDL is in this area's `schema.ts:136-146`, accessors are not).
- **Per-session container model (Area 3):** The two-DB session split, `processing_ack` state machine, `delivered` table, even/odd seq partition, `container_state` single-row table, `journal_mode=DELETE` mount-coherency requirement, and the live `destinations`/`session_routing` IPC are all only meaningful in the per-session container architecture. All marked SKIP-ARCH in the matrix; Area 3 owns the architectural decision.
- **Chat SDK bridge / adapter (Area 4):** `chat_sdk_*` tables and the `SqliteStateAdapter` (`src/state-sqlite.ts`) only matter if v1 ever adopts the Chat SDK — but the lock + TTL + queue primitives in there are independently useful (the matrix flags them ADOPT for the primitive, separate from any chat-sdk decision).
- **Agent-to-agent destinations module (Area 5):** `createMessagingGroupAgent` in this area calls into `agent_destinations` (`src/db/messaging-groups.ts:131-191`) with a `hasTable` guard — illustrative of the v2 module-table gating pattern. Any port of `messaging-group-agents` must coordinate with Area 5 on whether the destination side-effect comes along.
- **Scheduling (Area 3):** v1's `scheduled_tasks + task_run_logs` vs v2's `process_after/series_id/trigger=0` on `messages_in` — the choice between the two is a scheduling-design question owned by Area 3, not a pure-DB one. The DB-shape verdict (PORT v2's accumulator vs CONTRIBUTE v1's logs table) hinges on whichever scheduler wins.
- **Backup hook + JSON-snapshot IPC (Area 6):** The backup function lives in init.ts but the snapshot writers (`src/snapshots.ts`) are part of the file-IPC story owned by Area 6. The backup loop is small enough that it can be ported standalone; the snapshot writers cannot without rethinking IPC.

### Open questions

1. **Should v1 adopt the v2 numbered-migration framework even without the rest of the rewrite?** The framework is small (~70 LOC, `src/db/migrations/index.ts`) and self-contained. The blocking question: does v1 anticipate migrations driven by external skills/plugins? If yes, the `name`-keyed dedup is necessary; if no, the v1 array+sentinels system works fine. **Needs human verification of v1's plugin roadmap.**

2. **Should v1's `task_run_logs` be contributed upstream to v2?** v2 has scheduled-task functionality (via `messages_in` recurrence + `process_after`) but no run-log table. Adding one is a couple of indices + a JOINed view — but it conflicts with v2's "messages_in is the only source of truth for scheduled work" stance. Worth asking the upstream before opening a PR. **Needs human read on upstream design intent.**

3. **Does the v2 `hasTable` + module-migrations pattern make sense to backport into v1 at all?** v1's plugins use `Dockerfile.partial` + `mcp.json` + container-skills mounting (Area 5) — the central DB doesn't have a plugin-extension surface. If v1 plugins never need their own DB tables, the whole framework is unnecessary. **Needs human read on whether v1 plugins should grow DB tables.**

4. **The `chat_sdk_*` adapter's lock CAS pattern (`src/state-sqlite.ts:87-117`) — is the use case real for v1?** Chat SDK locks are about making sure only one container processes a thread at a time. v1 has `MAX_CONCURRENT_CONTAINERS` + per-group queues (Area 3) which enforce a similar invariant a different way. Porting the lock primitive is "cheap and could be useful" but might be solving a problem v1 doesn't have. **Needs human read on whether v1 has any unfulfilled lock/queue needs.**

5. **`container_state.tool_declared_timeout_ms` use case — is the equivalent on v1 a Bash hook?** v2 uses this single-row table to widen stuck-detection while a long-declared-timeout Bash is running (`src/db/session-db.ts:195-208`). v1's stuck detection lives entirely in `container-runner.ts` heartbeat checks. The tool-in-flight signaling is genuinely useful but the implementation is tied to v2's host-sweep loop. Worth verifying whether v1 has a stuck-bash-vs-long-bash false-positive problem before deciding to port. **Needs human read on operational pain.**

6. **Online backup retention=2 — too aggressive?** Two retained backups + one in-place WAL is conservative. In a single-tenant nanotars install with a few-MB DB, this is fine. If v2 adopts the backup loop (the CONTRIBUTE in the matrix), the retention should probably be higher because v2 has many session DBs. **Needs human read on operational sizing.**

7. **`getNewMessages`'s bot-prefix content backstop (`src/db/messages.ts:146-158`) — is the `is_bot_message` column actually populated reliably enough to drop the LIKE filter now?** Migration 004 backfilled it (`src/db/init.ts:117-122`) but uses the same `${botPrefix}:%` heuristic; if any pre-migration messages slipped through with a different prefix they'd still be in the table and an unguarded query would re-surface them. Risk is low — check by running a single SELECT — but worth a verification pass. **Needs human verification of historical data.**
